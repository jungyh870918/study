# [문서18] 시스템 디자인 완전 기초 — 모르면 안 되는 상식
> 강의에서 당연시하고 넘어간 기본기.

---

## 규모 추정 감각 (back-of-envelope)
- QPS = DAU × 인당 요청수 / 86400(초). 피크는 평균의 2~3배.
- 저장: 레코드 크기 × 개수. 1KB × 10억 = 1TB.
- 읽기:쓰기 비율 대개 100:1 이상(SNS) → 캐싱·복제가 답.
- 1일=86400초, 백만 QPS면 초당 백만 요청.

## 지연 숫자 감각 (아는 게 힘)
- 메모리 접근: ~100ns
- SSD 읽기: ~100μs
- 네트워크 같은 지역 왕복: ~0.5ms
- 디스크 탐색: ~10ms
- 대륙 간 왕복: ~150ms
- ★ 메모리는 디스크보다 ~10만 배 빠름 → 캐싱의 근거.

## 가용성 (nines)
- 99% = 연 3.65일 다운 / 99.9% = 8.7시간 / 99.99% = 52분 / 99.999% = 5분.
- SLA에서 "몇 개의 9"인지가 이중화 수준을 정함.

## 핵심 컴포넌트 역할
- 로드밸런서: 트래픽 분산.
- 리버스 프록시: 서버 앞단(TLS 종료, 캐싱, 라우팅).
- CDN: 정적 자원을 엣지에서.
- 캐시(Redis): 빠른 조회, 세션.
- 메시지 큐: 비동기·완충.
- DB: 영속 저장.
- 오브젝트 스토리지(S3): 파일·이미지·백업.

## 수평 확장의 전제
- 무상태(stateless) 서버: 상태를 서버에 두지 말고 외부화(세션→Redis).
- 그래야 아무 서버나 요청 처리 가능→LB로 분산.

## API 스타일
- REST: 자원 중심, 범용.
- gRPC: 서비스 간, 고성능.
- GraphQL: 클라가 필요한 것만.
- WebSocket: 양방향 실시간.

## 데이터 처리 방식
- 동기 vs 비동기: 즉시 응답 필요 vs 나중에 처리.
- 배치 vs 스트림: 모아서 처리 vs 실시간 흐름.
- OLTP(트랜잭션) vs OLAP(분석).

## 기본 보안
- 인증(누구냐) vs 인가(권한). JWT/세션.
- HTTPS/TLS(네트워크편). rate limiting.
- 최소 권한 원칙(IAM, 네트워크편).
