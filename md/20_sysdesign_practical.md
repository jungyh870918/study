# [문서20] 시스템 디자인 실전편 — 대표 설계 문제
> 자주 나오는 문제를 프레임워크(요구사항→규모→API→저장→HLD→병목)로.

---

## 1. URL 단축기 (TinyURL)
요구: 긴 URL→짧은 코드, 리다이렉트. 읽기 압도적(100:1+).
핵심:
- 코드 생성: 해시(충돌 처리) 또는 카운터+base62 인코딩(7자면 62^7≈3.5조).
- 저장: 키-값(단축코드→원본). 읽기 많으니 캐싱(Redis) 필수.
- 리다이렉트: 301(영구)vs302(임시). 클릭 추적 원하면 302.
- 규모: 읽기 폭발→CDN·캐시. 쓰기 적음→샤딩 거의 불필요.
트레이드오프: 해시(단순, 충돌) vs 카운터(충돌 없음, 분산 시 조율 필요→범위 분배).

## 2. 실시간 채팅
요구: 1:1/그룹, 실시간, 순서, presence, 오프라인 메시지.
핵심:
- 연결: WebSocket(양방향). 서버 여러 대→Redis Pub/Sub로 인스턴스 간 전파.
- 순서: 서버 시퀀스/타임스탬프.
- 전달: DB 저장, 재접속 시 커서로 놓친 것 복구(at-least-once).
- presence: Redis TTL+하트비트.
- 확장: 연결을 여러 서버 분산+상태 외부화. (네트워크 문서5와 연결)
트레이드오프: 강한 순서 보장 vs 지연. 그룹 대규모면 fan-out 비용.

## 3. 뉴스 피드 (SNS)
요구: 팔로우 기반 피드, 최신순/랭킹.
핵심 - fan-out 전략:
- Fan-out on write(push): 글 쓸 때 팔로워 피드에 미리 뿌림. 읽기 빠름, 셀럽(수백만 팔로워)이면 쓰기 폭발.
- Fan-out on read(pull): 읽을 때 팔로우 목록의 글을 모음. 쓰기 싸고 읽기 느림.
- ★ 하이브리드: 일반은 push, 셀럽은 pull(읽을 때 합침). 실무 정답.
- 캐싱: 피드를 Redis에. 랭킹은 점수 기반 정렬(힙/Sorted Set).
트레이드오프: 저장 공간(push 중복) vs 읽기 지연(pull).

## 4. 결제 시스템
요구: 정확성 최우선, 중복 방지, 외부 PG 연동.
핵심:
- 멱등성 키: 재시도로 이중 결제 방지(필수).
- 상태 머신: pending→paid→failed. 각 전이 기록.
- 큐: 결제 요청 완충, 순서 처리.
- PG 웹훅: 비동기 결제 결과 수신으로 최종 확정.
- 정합성: 강한 일관성(잔액·재고는 틀리면 사고). 트랜잭션.
트레이드오프: 정합성 위해 가용성·속도 일부 희생(CP 성향). (연상사전 결제와 연결)

## 5. 대용량 읽기 서비스 (일반 패턴)
- CDN(정적)→LB→무상태 서버(오토스케일)→캐시(Redis)→읽기 레플리카→마스터.
- 각 계층에서 걸러 원본 DB 부하 최소화.
- 쓰기는 마스터, 읽기는 레플리카. 캐시 무효화 주의.

## 6. 면접 팁 (경험 서술)
- 요구사항 질문부터: "동시 사용자? 읽기/쓰기 비율? 일관성 요구?"
- 규모를 숫자로: "DAU 100만, 읽기 100:1이면 캐싱이 핵심."
- 트레이드오프를 말로: "여기선 가용성을 위해 최종 일관성을 택하겠습니다."
- 병목을 스스로 짚기: "이 LB가 SPOF라 이중화하겠습니다."
- 과도한 설계 경계: "지금 규모엔 샤딩 불필요, 캐싱으로 충분."

## 7. 대량 트래픽 핸들링 — 계층별 방어 (종합)
핵심 원칙: 은탄환 없음. 요청이 DB(최고 비용)에 닿기 전 각 계층에서 걸러냄. 바깥일수록 싸고 안쪽일수록 비쌈.
방어선: [사용자]→①CDN→②LB→③Rate Limit→④앱서버(오토스케일)→⑤Redis 캐시→⑥메시지 큐→[DB 복제/샤딩]

### ① 캐싱 (가장 효과 큰 첫 방어선)
인과: 읽기 압도적(100:1)에서 같은 데이터 매번 DB 읽기는 낭비→캐시 반환으로 DB 부하 급감.
- CDN: 정적 자원 엣지에서. Redis: 동적·세션·집계. 앱 캐시: 메모리 값.
Redis 전략:
- cache-aside: 캐시 확인→없으면 DB→채움. 가장 흔함.
- ★ 무효화: 변경 시 캐시 삭제(안 하면 옛 데이터). 최다 실수.
- ★ Cache Stampede: 인기 키 동시 만료→수천 요청 DB로. 대응 TTL 지터/잠금(하나만 조회, 나머지 대기).
- Hot Key: 특정 키 집중→노드 과부하. 로컬 캐시/키 분산.
추가 활용: 세션 외부화(수평확장 전제), rate limit 카운터, 분산 락, Pub/Sub.

### ② 로드밸런싱 & 오토스케일링
LB: 여러 서버 분산. L4/L7, RR/least_conn/IP해시.
★ 수평 확장 전제=무상태(stateless): 상태(세션)를 서버에 두면 확장 막힘→외부화(Redis)해야 아무 서버나 처리.
오토스케일 두 층:
- HPA: CPU/메트릭 기준 Pod/인스턴스 수.
- Cluster Autoscaler/Karpenter: 노드 자체. AWS ASG: EC2 대수.
★ 실무:
- 예측형 vs 반응형: 반응형은 급증에 늦음(인스턴스 뜨는 시간)→예측 이벤트는 미리 예열(pre-warming).
- 스케일 아웃 빠르게, 인 천천히(다시 몰릴 수 있음).
- 콜드 스타트: 준비 전엔 트래픽 안 보냄(readiness probe).

### ③ Rate Limiting
인과: 어뷰징·봇·DDoS·폭주가 시스템 죽임→단위 시간당 제한.
- 토큰 버킷(버스트 허용, 가장 흔함)/누출 버킷/슬라이딩 윈도우.
- 분산은 Redis 카운터 공유. 초과 시 429+Retry-After.

### ④ 메시지 큐 (쓰기 완충)
인과: 읽기는 캐싱, 쓰기는 캐싱 불가(DB 반영 필요)→큐가 흡수해 DB 감당 속도로.
- 완충(backpressure), 비동기 분리(무거운 작업), 디커플링. Kafka/RabbitMQ.

### ⑤ DB 확장 (최후)
- 읽기 병목→읽기 복제+캐싱. 쓰기 병목→샤딩(조인·트랜잭션 잃어 최후).
- ★ 순서: 캐싱→읽기 복제→인덱스 최적화→샤딩.

### ⑥ 자원 모니터링 & 관측성 (대응의 눈)
인과: 안 보이면 대응 못 함. 병목 위치·스케일 시점 판단. 오토스케일도 이 메트릭 기반.
3축:
- 메트릭: QPS/지연/에러율/CPU·메모리/커넥션 풀. Prometheus+Grafana(HPA가 이걸로 스케일).
- 로그: 구조화+요청 ID. CloudWatch/Loki/ELK.
- 트레이싱: 분산 추적(요청 경로). Jaeger.
★ 반드시 보는 지표:
- p99 지연(평균보다 중요, 상위 1% 느리면 문제). p50/p95/p99 함께.
- 에러율(5xx 급증=위험), 포화도(saturation), 커넥션 풀 사용률(고갈→대기·타임아웃).
- 알림: 임계 넘으면 자동 알림→선제 대응.

### ⑦ 장애 격리 & 우아한 저하
- Circuit Breaker: 뒤 서비스 죽으면 빠르게 실패→연쇄 장애 방지.
- Graceful Degradation: 과부하 시 핵심만 유지(추천 끄고 주문만).
- Bulkhead: 자원 격벽으로 나눠 번짐 방지.
- Backpressure: 감당 못 하면 받는 속도 늦춤(큐/rate limit).

### 종합 시나리오 ("100배 급증 대응")
"계층별 방어선을 겹친다: CDN으로 정적 걷어내고, Redis로 읽기 부하 분리, 무상태 서버를 HPA·Karpenter로 오토스케일, 쓰기 폭주는 큐로 완충, 남용은 rate limit(429), DB 병목은 읽기 복제. Prometheus/Grafana로 p99·에러율 보며 대응, 뒤 장애는 circuit breaker로 격리, 예측 이벤트는 미리 예열."
한문장: "은탄환 없음. CDN·캐시·오토스케일·큐·rate limit·복제를 계층별로 겹치고, 모니터링으로 병목 보며 대응. 바깥일수록 싸고 DB가 최후."
