# [문서20] 시스템 디자인 실전편 — 대표 설계 문제
> 자주 나오는 문제를 프레임워크(요구사항→규모→API→저장→HLD→병목)로.

---

## 1. URL 단축기 (TinyURL)
요구: 긴 URL→짧은 코드, 리다이렉트. 읽기 압도적(100:1+).
핵심:
- 코드 생성: 해시(충돌 처리) 또는 카운터+base62 인코딩(7자면 62^7≈3.5조).
- 저장: 키-값(단축코드→원본). 읽기 많으니 캐싱(Redis) 필수.
- 리다이렉트: 301(영구)vs302(임시). 클릭 추적 원하면 302.
- 규모: 읽기 폭발→CDN·캐시. 쓰기 적음→샤딩 거의 불필요.
트레이드오프: 해시(단순, 충돌) vs 카운터(충돌 없음, 분산 시 조율 필요→범위 분배).

## 2. 실시간 채팅
요구: 1:1/그룹, 실시간, 순서, presence, 오프라인 메시지.
핵심:
- 연결: WebSocket(양방향). 서버 여러 대→Redis Pub/Sub로 인스턴스 간 전파.
- 순서: 서버 시퀀스/타임스탬프.
- 전달: DB 저장, 재접속 시 커서로 놓친 것 복구(at-least-once).
- presence: Redis TTL+하트비트.
- 확장: 연결을 여러 서버 분산+상태 외부화. (네트워크 문서5와 연결)
트레이드오프: 강한 순서 보장 vs 지연. 그룹 대규모면 fan-out 비용.

## 3. 뉴스 피드 (SNS)
요구: 팔로우 기반 피드, 최신순/랭킹.
핵심 - fan-out 전략:
- Fan-out on write(push): 글 쓸 때 팔로워 피드에 미리 뿌림. 읽기 빠름, 셀럽(수백만 팔로워)이면 쓰기 폭발.
- Fan-out on read(pull): 읽을 때 팔로우 목록의 글을 모음. 쓰기 싸고 읽기 느림.
- ★ 하이브리드: 일반은 push, 셀럽은 pull(읽을 때 합침). 실무 정답.
- 캐싱: 피드를 Redis에. 랭킹은 점수 기반 정렬(힙/Sorted Set).
트레이드오프: 저장 공간(push 중복) vs 읽기 지연(pull).

## 4. 결제 시스템
요구: 정확성 최우선, 중복 방지, 외부 PG 연동.
핵심:
- 멱등성 키: 재시도로 이중 결제 방지(필수).
- 상태 머신: pending→paid→failed. 각 전이 기록.
- 큐: 결제 요청 완충, 순서 처리.
- PG 웹훅: 비동기 결제 결과 수신으로 최종 확정.
- 정합성: 강한 일관성(잔액·재고는 틀리면 사고). 트랜잭션.
트레이드오프: 정합성 위해 가용성·속도 일부 희생(CP 성향). (연상사전 결제와 연결)

## 5. 대용량 읽기 서비스 (일반 패턴)
- CDN(정적)→LB→무상태 서버(오토스케일)→캐시(Redis)→읽기 레플리카→마스터.
- 각 계층에서 걸러 원본 DB 부하 최소화.
- 쓰기는 마스터, 읽기는 레플리카. 캐시 무효화 주의.

## 6. 면접 팁 (경험 서술)
- 요구사항 질문부터: "동시 사용자? 읽기/쓰기 비율? 일관성 요구?"
- 규모를 숫자로: "DAU 100만, 읽기 100:1이면 캐싱이 핵심."
- 트레이드오프를 말로: "여기선 가용성을 위해 최종 일관성을 택하겠습니다."
- 병목을 스스로 짚기: "이 LB가 SPOF라 이중화하겠습니다."
- 과도한 설계 경계: "지금 규모엔 샤딩 불필요, 캐싱으로 충분."
