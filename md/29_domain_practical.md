# [문서29] 도메인 설계 사례 — 실무 경험을 면접 언어로
> 복잡한 비즈니스 로직을 설계·구현한 경험. 공통 뼈대=상태 머신+권한+큐/멱등성+감사.

---

## A. 구독 결제 시스템
전체: 결제는 정확성 최우선+외부 PG 의존+네트워크 실패 필연→큐·재시도·멱등성·상태 머신 필요.

### 카드 등록 & PG 모듈
★ 카드 정보 직접 저장 안 함: 카드번호 DB 저장=PCI-DSS 위반+유출 재앙.
- PG(토스/아임포트/스트라이프)에 카드 등록→빌링키(billing key)만 저장. 결제는 "이 빌링키로 N원".
- 우리는 빌링키+카드 마지막 4자리(표시용)만.

### 결제 큐 & 재시도
- 왜 큐: 정기 결제 몰림(월초)→PG rate limit·DB 부하→큐로 완충. 비동기로 응답 빠름.
- ★ 멱등성 필수: 재시도 시 이중 결제 방지. 멱등성 키(주문 ID)로 "이미 처리?" 확인. PG도 멱등성 키 지원.
- 재시도: 지수 백오프(1분→5분→30분, 즉시 반복은 PG 차단). N회 실패→past_due+알림. DLQ로 격리.

### 환불 큐
- 부분/전액 환불도 큐. 멱등성(환불 ID). 상태 추적(요청→처리중→완료/실패), 웹훅으로 확정.

### 구독 상태 머신 (제일 중요)
상태: active / cancel_scheduled(취소 예정, 기간 말까지 사용) / canceled / past_due(연체) / paused.
★ "취소 예정"이 왜: 취소해도 이미 낸 기간까진 사용→즉시 canceled 아니라 cancel_scheduled, 크론이 기간 말에 canceled 전환. "취소했는데 왜 쓸 수 있냐/왜 결제됐냐" CS 방지.

### 크론잡 (매일 자정)
1. cancel_scheduled 중 만료 지난 것→canceled
2. active 중 결제일 도래→결제 큐 투입
3. past_due 유예 지난 것→서비스 중단
4. 곧 만료→알림
★ 주의: 멱등성(중복 실행돼도 이중 처리 X, "오늘 처리했나" 체크), 대량은 배치 분할, 실패 격리.

### 시트 관리 & 할인
- 기본 제공 시트+추가는 결제. 시트 조정: 늘리면 즉시 과금(비례), 줄이면 다음 주기.
- ★ proration(일할 계산): 월 중간 추가 시 남은 기간만 과금("15일 추가=반달치").
- 할인 정책: 쿠폰/프로모션/연간 할인을 정책으로 분리(하드코딩 금지). 적용 순서·중복 규칙.

### 히스토리
- 구매/환불 히스토리(불변 로그): 정산·CS·분쟁. 상태 변경도 로그.

경험 서술: "구독을 상태 머신으로. 취소는 cancel_scheduled로 두고 크론이 기간 말 전환→CS 제거. 결제는 큐+멱등성 키로 이중 결제 막고 지수 백오프 재시도. 카드는 PG 빌링키로만 저장해 PCI 부담 제거. 시트 추가는 proration 일할 계산."

## B. 건설 감리 결재 시스템
도메인: 건설사(시공)-감리사(감독)가 문서로 소통, 결재 체계로 진행, 공정률·집행률 계산·비교. 권한 제어+문서 관리 핵심.

### 프로젝트 & 주체 등록
- 프로젝트 생성→건설사 등록(시공)+감리사 등록(감독). 주체별 권한 다름(건설사=작성, 감리사=검토·승인).

### 공정률 & 집행률 (핵심 로직)
- 공정률: 실제 시공 진척(%). 집행률: 예산 대비 집행된 시공비용(%).
- ★ 비교: 집행률≫공정률=돈은 썼는데 진척 안 됨=위험 신호(부실·과다 청구). 감리사가 판별. 시스템이 두 수치 계산·비교가 핵심 가치.

### 감리사 판별 & 최종 채택
- 여러 감리사 판별 비교(같은 항목 다른 의견 가능)→최종 채택(이견 조율해 하나로 확정). "누구 판단을 최종으로" 워크플로우.

### 결재 체계 (상태 머신+권한)
- 문서 상태: draft→submitted→under_review→approved/rejected. 반려되면 draft로.
- 각 전이에 권한: 작성자만 submit, 감리사만 approve. 결재선을 데이터로 관리. 도장 단계별 날인.

### 문서 코드 관리
- ★ 체계적 코드 필수(문서 수천 개): 프로젝트-공종-종류-일련번호(예: PRJ001-CIVIL-INSP-0042).
- 권한별 코드: 건설사·감리사가 보는/만드는 코드 범위 다름. 코드로 검색·분류·이력. 도면 개정(rev) 관리.

### 권한 제어 (RBAC)
- ★ 문서별 작성·수정 권한: 역할(건설사/감리사/관리자)별.
- 문서 종류·상태별: 건설사는 작성, 감리사는 검토·승인. approved는 불변(수정 불가). 반려된 것만 재수정.
- 도장 등록: 각 주체 도장 등록, 승인 시 권한 확인 후 날인.

경험 서술: "건설사-감리사가 결재 워크플로우로 소통. 문서를 상태 머신(작성→제출→검토→승인)으로, 각 전이에 RBAC로 건설사는 작성만·감리사는 승인만. 승인 문서는 불변 잠금, 반려는 재작성만. 공정률·집행률 계산·비교해 집행이 진척보다 앞서면 감리사가 위험 판별, 여러 의견 비교해 최종 채택. 문서는 프로젝트-공종-종류-일련번호 코드 체계+권한별 접근."

## 두 도메인 공통 5년차 역량 (★ 면접 포인트)
완전히 다른 도메인이지만 같은 뼈대로 푼다:
- 상태 머신: 구독 상태 / 문서 결재 상태. "상태 명확히 정의+전이 통제"가 복잡 로직의 핵심.
- 권한 제어(RBAC): 시트 권한 / 문서 작성·승인 권한.
- 큐+재시도+멱등성: 결제·환불 / 대량 처리.
- 크론잡: 구독 만료 체크 / 정기 리포트.
- 감사 로그: 결제 히스토리 / 문서 이력.
- 정책 분리: 할인 정책 / 결재선 규칙을 하드코딩 않고 데이터로.
★ 메시지: "상태 머신+권한+큐/멱등성+감사"라는 같은 뼈대로 다른 도메인을 푼다. 이 패턴을 아는 게 '복잡한 비즈니스 로직을 다뤄봤다'의 실체.

## C. 병원 결제 시스템 (멀티테넌트 + 크로스디바이스)

### 토스페이먼츠 PG (요청-승인 2단계)
★ 핵심: 요청과 승인이 분리된 2단계(정합성 위해).
1. SDK 결제 요청(orderId, successUrl, failUrl)
2. 카드사 인증창→사용자 인증
3. 성공→토스가 paymentKey 발급+successUrl 리다이렉트(쿼리에 paymentKey/orderId/amount)
4. 백엔드가 받은 값이 요청 때 값과 일치하는지 검증
5. 백엔드가 결제 승인 API(paymentKey+시크릿 키)→실제 청구
6. 승인 완료→Payment 객체→paymentKey DB 저장
★ 왜 2단계: 한 번에 하면 웹훅 의존→불일치 위험. 분리하면 백엔드가 승인 직전 금액 재검증(위변조 방지). "프론트 금액 그대로 안 믿는다". paymentKey는 DB 저장(취소·조회), 시크릿 없으면 무용이라 노출돼도 안전.
상태: READY→IN_PROGRESS→DONE.

### 멀티테넌트 (서브도메인)
- 메인 도메인 구매→병원별 서브도메인(hospital-a.service.com→A병원).
- 테넌트 식별: 서브도메인으로 병원 판별→그 병원 데이터만.
- 데이터 격리: 테넌트 ID로 모든 쿼리 필터(또는 스키마/DB 분리).
- 격리 트레이드오프: 공유 DB+테넌트 ID(간단·저비용·약한 격리) vs 스키마/DB 분리(강한 격리·고비용). 대개 공유 DB+테넌트 ID로 시작.

### 크로스디바이스 결제 (QR+문자)
시나리오: 데스크톱에서 진료 확인, 결제는 환자 폰.
1. 데스크톱서 결제 정보 확정→결제 URL 생성
2. 전달(옵션): QR 코드(대면, 폰 스캔) or 문자 SMS(원격)
3. 환자 폰 브라우저서 결제 URL→토스 결제
4. 완료→영수증
★ 결제 URL에 토큰(세션 식별, 만료·일회성). 상태 동기화(폰 결제 끝나면 데스크톱 갱신, 폴링/웹소켓).

### 단계별 진행 바 (UX)
- 차트번호 입력→수술 종류 선택→결제 정보 재확인→결제→영수증.
- ★ 결제 정보 재확인: 직전 금액·항목 재표시(되돌리기 어려우니 실수 방지). 진행 바로 이탈 감소.

### JSON 대량 업로드
- 병원별 라식 수술 종류를 JSON→스크립트 벌크 인서트.
- ★ 멱등성(재실행 시 중복 X, upsert), 업로드 전 스키마 검증.

### 스택 (Neon+Vercel+Next+Lucide+Drawer)
- Neon: 서버리스 PostgreSQL(사용량 기반, 자동 확장, 브랜칭).
- Vercel: Next 배포(서버리스·엣지, Git 자동 배포).
- Next: 서버 컴포넌트·서브도메인 라우팅. Lucide: 가벼운 아이콘. Drawer: 모바일 하단 슬라이드 UI.
★ 서버리스(Neon)+서버리스 배포(Vercel)=인프라 관리 없이 서비스, 트래픽 없으면 비용 최소.

경험 서술: "병원별 멀티테넌트를 서브도메인 격리, 토스는 요청-승인 2단계로 백엔드에서 금액 재검증. 데스크톱 확정 결제를 QR/문자로 환자 폰에 넘기는 크로스디바이스. paymentKey 저장해 취소·조회 대비. 수술 종류는 JSON+스크립트 멱등 벌크 업로드. Neon+Vercel 서버리스로 인프라 부담 없이."

## D. 출구조사 데이터 정규화 (더러운 외부 데이터 길들이기)

### 문제 — flat한 엉망 데이터
선관위 원본이 계층이어야 할 것을 같은 레벨에 나열:
{ 선거구코드, 지역, 연령10, 연령20...연령60, 성별남, 성별여 }
연령대·성별이 상위 카테고리 하위여야 하는데 전부 최상위 키→처리 어렵고 키 이름 조금 바뀌면 코드 깨짐.

### 해결 — 정규화
flat→계층 파싱:
{ 선거구코드, 지역, 연령:{10,20,30...}, 성별:{남,여} }
- 공통 패턴(연령XX, 성별X)을 파싱해 상위 카테고리로 그룹핑.
- 리액트식 정규화: 일관된 객체 구조로 렌더링·조작 단순화.

### Enum화
- 컬러 코드·카테고리를 enum 중앙 관리. 정당별 색 한 곳에서, 매직 넘버·스트링 제거.

### ★ 핵심 성과 — 변경에 강한 구조
"원본 키 이름이 조금 바뀌어도 대응, 데이터 로드 오류 없음"이 핵심 가치:
- 파싱 레이어로 원본과 내부 구조 분리→원본 바뀌어도 파싱 규칙만 조정, 내부 코드 그대로.
- 패턴 기반 파싱(연령\d+ 같은 패턴)→이름 조금 바뀌어도 흡수.
- 방어적 로딩→예상 못 한 필드·누락에도 안 깨짐.
= DI("인터페이스 의존, 구현 변화에 강하게")+관심사 분리의 데이터 버전.

### 면접 가치
단순 구현 아닌 설계 판단의 증거: 더러운 데이터 길들이기(현실 데이터는 항상 엉망)+변경에 강한 설계(원본 변경 흡수)+정규화·enum 기본기.
경험 서술: "출구조사 원본이 연령·성별을 flat 나열해 처리 어려움. 공통 패턴 파싱해 상위 카테고리 하위로 정규화, 컬러는 enum 중앙화. 핵심은 파싱 레이어로 원본과 내부 구조 분리라, 원본 키 바뀌어도 파싱 규칙만 조정하면 돼 로드가 안정적."

## 공통 확장 (C·D 추가 역량)
- 멀티테넌시: 서브도메인 격리+테넌트 ID 필터.
- 외부 연동 정합성: PG 요청-승인 분리, 백엔드 재검증("프론트 값 안 믿기").
- 크로스디바이스: URL 토큰+상태 동기화.
- 데이터 파이프라인: 파싱 레이어로 원본과 내부 분리→변경에 강하게(정규화·enum·방어적 로딩).
- 벌크 처리: JSON+스크립트+멱등성.
★ 메시지: "외부(PG·선관위)는 못 믿고 변한다. 경계에 검증·파싱 레이어를 둬서 내부를 보호한다"가 두 사례의 공통 교훈.

## E. 행안부 SSO(AnyID) 연동 — 제약 환경 극복
프레이밍: "SSO 붙였다"가 아니라 "폐쇄망·문서 부족·벤더 모듈·운영 테스트 제약에서 정부 인증을 끝까지 연동". 문제 해결력·끈기의 최상급 사례. 난관을 어떻게 우회했는지가 핵심.

### 정상 인증 흐름
1. 로그인 버튼→API 트리거
2. 통합 세션 활성화 확인: 있으면 바로 로그인 전환 / 없으면 로그인 화면 렌더링+tx 값 수신
3. tx 값으로 AnyID 서버 인증 요청
4. 위조 요청 필터 통과
5. 성공→콜백 클래스 함수 실행
6. 유저 정보 파싱→세션 저장
★ 핵심 분기: 통합 세션 있으면 바로, 없으면 화면+tx. 이 옵션 구현이 개발 환경에선 불가능했던 게 첫 난관.

### 난관1 — 폐쇄망(인바운드/아웃바운드 룰)
- AnyID 인증은 우리 서버↔AnyID 양방향 통신 필요. 개발 서버는 방화벽 룰 안 열려 요청도 콜백도 불가→인증 흐름 재현 불가.
- (네트워크편 인바운드/아웃바운드·방화벽이 실전에서 발목 잡은 사례.)
- 우회: 인증 성공 리턴값을 JS로 흉내(모킹)→개발 서버서 인증 함수 순차 실행 흉내. 흐름 검증용 스텁(위조 필터·실제 콜백은 검증 못 함, 한계 명확).

### 난관2 — 운영 서버 직접 테스트(극약 처방)
- 개발 서버로 실제 인증 못 끝내→주말에 운영 서버에 히든 로그인 페이지 배포해 테스트.
- 운영이어야 방화벽·통합 세션·실제 연동 작동. 히든 페이지(메인에 등록하되 미노출)로 콜백 클래스 함수 돌려 유저 정보 파싱→세션 저장 검증. 주말+히든으로 영향 최소화.
- 면접 서술: "위험하지만 폐쇄망 제약상 실제 왕복 재현할 곳이 거기뿐. 트래픽 적은 주말+미노출 히든 페이지로 영향 최소화하며 검증." = 리스크 인지·통제.

### 벤더 모듈 특성
- AnyID 모듈=서블릿 자동 등록(URL 지정 안 해도 해당 주소 요청 시 백엔드 동작). 프론트 모듈·콜백 클래스 함수 제공.
- 편함: 보일러플레이트 없음. ★ 함정: 내가 제어 못 하는 부분 많아 문제 시 내부 못 봐 디버깅 어려움. 벤더 모듈의 전형적 트레이드오프.

### 난관3 — JBoss 배포 사고(프론트 누락)
- JBoss라 빌드 산출물(WAR/EAR, Java 웹 아티팩트) 풀어 배포. 백엔드+Nexacro 프론트가 한 패키지.
- 빌드에서 Nexacro 프론트 빠진 채 배포→사이트 깨짐. 해결: 백엔드만 부분 배포+서버 재시작.
- ★ 교훈: 백/프론트 한 아티팩트로 묶이면 부분 배포 어렵고 사고 시 전체 깨짐. (CI/CD편 "레포 분리·아티팩트 관리"의 반면교사.)

### 도메인 복잡성 — 다중 계정 & 사업자 유형
★ 타 민원과 다름: 한 사람이 여러 계정 가능→단순 자동 연동 안 됨.
- 흐름: 인증→개인 식별→그 정보로 유저 정보 재검색→여러 계정 프론트로 리턴→사용자가 계정 선택→로그인.
- 마이페이지 AnyID 연동/해제.
- 사업자 유형별: 개인사업자/법인/개인 각각 다르게.
- ★ CI/DI 분리: CI(연계정보, 개인 식별 고유값, 사이트 무관 동일) / DI(중복가입확인정보, 사이트 내 중복 방지, 사이트마다 다름). 사업자 유형 따라 따로 관리.
- 개인정보 격리: 개인·기업 식별정보를 세션 저장해 외부로 안 돌게 통제.

### 난관4 — 조직 간 협업(광주 서버 센터)
- 폐쇄망 방화벽 룰을 광주 서버관리센터에 직접 요청. 내 서버 IP만 주고 센터 NATS 정보 안 알려줘 통신 문제.
- 타 사이트서 민원 처리→내 사이트로 넘어와 리다이렉트하는 사이트 간 연계까지.
- ★ 교훈: 대형 정부 시스템은 기술만이 아니라 조직 간 협업·정보 공유가 병목. "누가 무슨 정보 가졌고 누구에게 요청" 파악도 실력.

### 면접 자산화
나쁜 서술: "방화벽 때문에 고생." 좋은 서술(문제→제약→판단→결과):
"행안부 AnyID를 연동했는데 폐쇄망 방화벽으로 개발 서버서 인증 왕복 재현 불가. 처음엔 리턴값을 JS 모킹해 흐름 검증, 실제 콜백·위조 필터는 트래픽 적은 주말 운영 서버 히든 페이지로 리스크 통제하며 진행. 한 사람이 여러 계정 가능해 인증 후 CI로 유저 재검색해 계정 선택 단계 추가, 개인·개인사업자·법인별 CI/DI 분리. 배포는 백/Nexacro 프론트가 한 아티팩트라 프론트 누락 사고 후 부분 배포로 대응. 방화벽 룰은 광주 센터 협업 필요했는데 정보 공유 병목 겪으며 조직 협업도 기술만큼 중요함을 배움."
강한 이유: 제약 하 판단(모킹→운영 테스트, 리스크 인지)+도메인 복잡성(다중 계정, CI/DI)+실패 숨기지 않음(배포 사고)+비기술 현실(조직 협업)=성숙함.

## 도메인 사례 총정리 (A~E 관통 원칙)
- 상태 머신: 구독/문서 결재/결제 상태.
- 권한 제어(RBAC): 시트/문서/계정.
- 큐+멱등성: 결제·환불.
- 경계 방어: PG 금액 백엔드 재검증, 출구조사 파싱 레이어, SSO 위조 필터. "외부는 못 믿고 변한다→경계에 검증·파싱 레이어."
- 제약 하 판단: 폐쇄망에서 모킹→운영 테스트(리스크 통제).
- 데이터 정규화: 출구조사 flat→계층, CI/DI 분리.
★ 최종 메시지: "도메인은 달라도 상태 머신+권한+큐/멱등성+경계 방어라는 뼈대로 푼다. 제약이 있으면 리스크를 인지·통제하며 우회한다." = 5년차 설계·문제해결 역량.
