# [문서0] 설명이 뭉개질 때 — 답변 기술
> "말로는 아는데 깔끔하게 못 끊을 때" 대응. 모든 주제 공통.

## 문제의 정체
지식이 없는 게 아니라 "구조(뼈대)"가 없는 것. 꺼내는 순서가 없어 곁가지로 샘.
면접관은 여기서 "이해 vs 주워들음"을 판단.

## 핵심 해결: 3단 템플릿에 강제로 끼우기
① 정의(한 문장): "X는 ~하는 것입니다."
② 왜/구조(핵심 1~2개): "핵심은 ~때문이고, ~로 동작합니다."
③ 실무/문제(구체 1개): "실무에선 ~할 때 쓰고, ~가 문제될 수 있습니다."
→ 이 문서 시리즈의 "정의→핵심→문제/인과" 구조가 곧 답변 대본.

## 실전 테크닉 5
1. "먼저 결론부터 말하면"으로 시작 → 정의 먼저 던지면 절반은 깔끔.
2. 숫자 선언: "크게 세 가지입니다" → 뇌가 스스로 정렬. 2개만 나와도 "세번째는 지금 안 떠오릅니다"로 정직.
3. 대비 구조: "A는 이렇고 반면 B는 저렇다" → 가장 안 뭉개지는 화법. (TIME/CLOSE_WAIT, 흐름/혼잡제어, 포워드/리버스)
4. 막히면 축소: 전부 설명하다 뭉개지느니 "A,B는 확실, C는 개념만" 범위 긋기. 솔직함=신뢰.
5. 모르면 접근법 전환: "정확힌 기억 안 나지만 저라면 이렇게 확인하겠습니다: ~" → 정답보다 문제해결 방식을 봄.

## 연습법
문서4(확인질문)를 소리 내어, 반드시 3단 템플릿으로 답하고 녹음→재생. 새는 지점이 보임.
"정의 한 문장 → 핵심 → 실무" 리듬이 몸에 배면 안 뭉개짐.

## 학습·면접 관점: "핵심이 튼튼하면 도구는 쉽다"
주장: 제품 출시 지향 개발자에게 중요한 건 깊은 도구 지식이 아니라, 튼튼한 핵심(네트워크·DB·알고리즘)+도구를 쉽게 가져다 쓰는 능력. 프레임워크는 프레임워크일 뿐.
맞는 이유:
- 프레임워크·클라우드는 몇 년마다 바뀜(Express→Nest, EC2→EKS→서버리스). 도구 지식은 감가상각.
- 네트워크(TCP/HTTP)·DB(인덱스/트랜잭션)·알고리즘(복잡도)은 20년째 그대로. 감가상각 안 되는 자산.
- 핵심 튼튼하면 새 도구는 "그 개념의 구현이구나"로 빠르게 습득(Nest DI=스프링, K8s 오토스케일=그냥 오토스케일).
면접에서 강한 이유:
- "도구 얼마나 아느냐"로 겨루면 전문가한테 짐.
- "핵심 알고 도구 빠르게 흡수"=어떤 도구 와도 제품 낸다는 신호→채용 본질에 부합.
★ 균형(5년차 미묘함):
- "도구는 도구일 뿐"이 안 배워도 된다는 뜻 아님. 기본·중요 개념은 알아야 쉽게 씀.
- 한두 개는 깊게 파본 경험이 있어야 "필요하면 깊이 판다"를 증명. 전부 얕으면 겉핥기로 보임.
- 이상적: "핵심은 깊게, 도구는 넓고 얕게+한두 개는 깊이 파본 흔적."
학습 설계와 일치: 네트워크·DB·알고리즘=핵심 깊게 / AWS·K8s·프레임워크=도구 실용적으로 / 기술 연상 사전=빠른 흡수 반사.
