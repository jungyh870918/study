# [문서30] 프로젝트 Voyage — 설명 자료 (개요)
> "무엇을 만들었나"보다 "왜 그렇게 설계했나 / 어떤 문제를 어떻게 풀었나".
> 면접관이 좋아하는 건 기능 나열이 아니라 판단의 근거. 각 항목 "문제→선택→이유→결과".

---

## 1. 한 문장 소개
Voyage = 실제 상황 대화문을 7단계 반복 청취로 내재화하는 오디오 우선(audio-first) 영어 회화 학습 앱.
- 웹/iOS/Android를 단일 코드베이스(Capacitor)로 서비스.
- 콘텐츠·음원·삽화 3개 도메인의 생성 파이프라인을 직접 설계.

엘리베이터 피치(30초):
"사람은 문장이 아니라 장면을 기억한다는 원리에서 출발한 앱. AI로 대화 콘텐츠를 대량 생성하되 '들으면 장면이 그려지는 문장'만 통과시키는 품질 게이트를 만들었고, TTS 음원·삽화까지 파이프라인으로 자동화. iOS는 앱스토어 라이브, Android는 심사 제출 완료."

## 2. 기술 스택 (왜 골랐나까지)
- 백엔드 FastAPI(Python): 콘텐츠·채점·음원 파이프라인이 전부 Python 생태계(LLM SDK, 오디오 처리)라 언어 통일.
- DB PostgreSQL(Railway 메인/Neon 폴백): 운영 안정성 위해 메인-폴백 이중화. 로컬은 SQLite.
- 프론트 React+Vite+Capacitor: 웹/iOS/Android를 하나의 코드베이스로. 개인 리소스로 3플랫폼 커버하는 현실적 선택.
- TTS ElevenLabs/Edge-TTS: 음질(ElevenLabs) vs 비용(Edge) 트레이드오프를 캐시로 흡수.
- 스토리지 Cloudflare R2: 음원은 DB가 아니라 R2에, 백엔드가 302 리다이렉트 → DB 경량화.
- 인증 Supabase Auth(이메일 OTP): 로그인 없이 2회 무료 체험 → 진입장벽 낮춤.
- 배포 Railway(백엔드)/Vercel(랜딩): 백엔드는 git push 즉시 배포, 프론트는 앱 빌드 필요(이 차이가 배포 전략 핵심).

## 3. 아키텍처 — 핵심 데이터 흐름
```
콘텐츠:  batches/*.json → content.py(코드 내장 단일소스) → migrate_content.py → DB
음원:    generate_all.py → ElevenLabs TTS → 캐시 → R2 (백엔드는 R2로 302 리다이렉트)
프론트:  React/Capacitor ─apiFetch→ FastAPI(main.py) ─→ DB조회 + R2
인증:    Supabase Auth(JWT) ─Bearer→ 서버에서만 검증
```
강조할 설계 결정:
- 단일 소스 원칙(SSOT): 콘텐츠는 반드시 batches JSON→content.py→migrate→DB 경로로만. DB를 직접 수정 안 함. 이유: DB 직접 고치면 다음 migrate 때 덮어써져 유실(→8번 사례).
- 텍스트/음원 파이프라인 분리: 텍스트는 즉시 배포(migrate), 음원은 별도 파이프라인(과금 이슈). 관심사 분리.

## 4. 가장 자랑할 것 — AI 콘텐츠 생성 + 품질 게이트
가장 임팩트 있는 부분. "AI로 대량 생성하면 품질이 무너진다"를 시스템으로 풀었다.
문제: LLM으로 레슨 대화 대량 생성하면 문법은 맞지만 "외워지지 않는 문장"이 쏟아짐. 반복 암기 앱엔 치명적.
해결 — 단 하나의 판별 기준:
"이 문장을 들으면 머릿속에 장면이 그려지는가?"
- ✅ "Why didn't he pass?" / "My feet are killing me."
- ❌ "This is getting tense." / "It smells amazing."
2층 검수 시스템:
- 1층 구조: 스키마·필드 개수·orphan·중복 → validate.py(기계적), 위반 0건.
- 2층 품질: 장면·감정·난이도·자연스러움 → 검수 에이전트/사람(판단), 체크리스트 통과.
★ 핵심 인사이트: "구조 통과 ≠ 품질 통과". 기계는 필드 맞는지만, 좋은 콘텐츠인지는 못 봄 → 두 층 분리.
생성/검증 주체 분리:
- 생성은 서브에이전트가 병렬로.
- 검증은 반드시 다른 주체가 교차검증(공통 맹점 회피).
- 애매한 자연스러움만 외부 LLM(GPT/Gemini)에 독립 자문 — 전수 투입은 과함.

## 5. 채점 시스템 — 임베딩 + 메타데이터 보정
"가장 어려웠던 기술 문제"로 쓰기 좋음.
문제: 발화 채점은 cos 유사도(75%)+표면 매칭(25%) 하이브리드인데 임베딩만으론:
- "세련된 정답"(다른 표현이지만 맞음)을 오답 처리
- "구조는 비슷한 오답"을 정답 처리
해결 — 두 종류 메타데이터 주입:
- references(패러프레이즈): 정답+모든 references와의 cos 최댓값 사용 → 다르게 답해도 등록된 패러프레이즈와 가까우면 구제(관대).
- critical_keywords(핵심어): 핵심어 전부 빠진 구조유사 발화에 점수 상한(cap) → 오답 차단(변별).
★ 한쪽은 위로(구제), 한쪽은 누름(차단). 양방향 보정이 설계 핵심.
안전장치: "틀린 메타보다 빈 메타가 안전하다."
- 확신 없는 references는 비움(빈 것도 채점 정상, 구제만 없음).
- 부정확한 references는 오답을 정답으로, 부정확한 keywords는 정답을 오답으로 → 생성+교차검증 필수.

## 6. 실제 디버깅 — STT 포맷 이슈 (플랫폼별 함정)
"플랫폼 차이 미묘한 버그를 어떻게 잡았나". 구체적이라 신뢰를 줌.
증상: Android에서만 발음 채점이 "잘 안 들렸어요"로 깨짐. iOS 정상.
원인: 녹음 라이브러리가 Android에서 raw AAC(ADTS)를 m4a 라벨 붙여 전송. Whisper는 aac 미지원→빈 텍스트. iOS는 실제 m4a라 통과 → 플랫폼별로 다르게 깨짐.
해결:
- 백엔드에서 매직바이트로 실제 컨테이너 감지(파일 라벨 안 믿음).
- 미지원이면 ffmpeg로 wav(16kHz mono) 변환.
- ADTS를 MP3보다 먼저 엄격 검사(0xFF 동기 바이트 충돌 버그도 수정).
★ 교훈: "클라이언트가 보내는 메타데이터(파일 라벨)를 믿지 말고 실제 바이트로 검증하라." 실기기 확인 완료.

## 7. 배포 전략 — "짝/의존성" 관리
릴리스 엔지니어링 감각. 코드만 짜는 게 아니라 배포까지 책임진다는 인상.
서버 배포 vs 앱 빌드 차이:
- 백엔드/데이터: git push → Railway 자동 → 즉시 반영.
- 프론트(jsx/css): 앱 빌드 필요 → 여러 수정 모아 한 버전(1.0.2)에.
★ 핵심 — "같이 가야 하는 짝(의존성)" 관리:
대표 사례: 음원에서 한국어 낭독 제거 ↔ 미리보기 모달 추가
- 이 둘은 반드시 동시에 나가야 함.
- 음원만 먼저 빼면 → 기존 사용자는 한국어 안내가 사라지는데 대체 모달도 없음 = UX 훼손.
- 배경음 차단과 대조: 배경음은 "빼기만" 하므로 짝 없어 서버만으로 처리 가능. intro_kr은 "빼고 모달로 대체"라 짝 필수.
말할 포인트: "각 변경이 독립적인지, 짝으로 묶여야 하는지 구분하는 게 배포 사고의 절반."

## 8. 실전에서 배운 것 — 파이프라인으로 구조적 해결
성장 서사로 쓰기 좋음.
상황: 한 레슨에 대화 2개 합쳐진 걸 독립 레슨으로 분리하니, 분리 쌍의 title·썸네일이 동일 복제되어 화면 구분 안 됨(54쌍, 108개 전부).
잘못 vs 올바른 접근:
- ❌ 임시: Railway DB 직접 UPDATE → 다음 migrate 때 덮어써짐.
- ✅ 근본: batch 원본 수정→content.py 재조립→migrate 재실행. 소스와 DB 정합성 확보.
스스로 판단해 해결한 것: 분리본(L79~123)이 전부 카테고리 "기타"로 떨어지는 버그 발견 → detect_category에 번호−45로 원본 짝 카테고리 물려받는 로직 추가 → "기타" 0건.
말할 포인트: "증상을 DB에서 직접 고치면 편하지만, 단일 소스 원칙을 지키려 파이프라인 위에서 고쳤다. 당장은 손이 더 가도 재현성·정합성 확보."

## 9. 시스템 설계 판단들 (짧게 여러 개)
"이 프로젝트에서 배운 설계 원칙?"에 던질 카드:
- 관심사 분리: 텍스트/음원/삽화를 독립 도메인 파이프라인으로. 각자 생성 도구·비용 구조 달라서.
- 비용을 캐시로 흡수: TTS 캐시 키=text+voice+rate. 구조·GAP 변경은 무료 재생성, 새 문장·화자만 과금. 캐시 키 설계로 비용 통제.
- 폴백 우선(graceful degradation): lesson_id 없이 요청 와도 에러 없이 기본 채점. 앱 구버전 호환.
- 단일 진실원(SSOT): validate.py가 스키마의 정답. 문서와 코드 충돌 시 코드가 이김.
- 메인-폴백 DB 이중화: Railway 메인/Neon 폴백. 단 "migrate가 쓴 DB와 앱이 읽는 DB가 같은지" 항상 확인(동기화 함정).

## 10. 예상 질문 & 대비
- "왜 Capacitor?" → 개인/소규모로 3플랫폼. 네이티브 성능 필수 아니라 웹뷰로 충분. 단 웹뷰 한계(iOS <audio>.volume 무시 등)를 서버 토글로 우회한 경험도.
- "AI가 만든 콘텐츠 품질을 어떻게 믿나?" → 2층 검수(구조 validate+품질 게이트)+생성/검증 주체 분리+애매한 것만 외부 LLM. "장면이 보이는가" 단일 기준.
- "가장 어려웠던 문제는?" → STT 플랫폼별 포맷(6번) 또는 채점 메타 보정(5번).
- "채점이 틀리면?" → "틀린 메타보다 빈 메타가 안전". 확신 없으면 비워서 오채점 방지.
- "혼자 다 했나/협업은?" → 역할 분담: 판단·명세·코드는 메인, 대량 생성·검수는 서브에이전트 병렬, 결정·빌드·DB 반영은 사람. 교차검증.
- "규모는?" → 레슨 124개+(E/L/S 3난이도), iOS 앱스토어 라이브, Android 심사 제출 완료.

## 11. 30초 / 2분 / 5분 버전 가이드
- 30초: 1번 엘리베이터 피치.
- 2분: 1번 + 4번(품질 게이트) + 성과(iOS 라이브/Android 제출).
- 5분: 3번(아키텍처)→4번(콘텐츠 파이프라인)→5번 또는 6번(기술 문제 하나)→7번(배포 짝)→9번 원칙 1~2개.

## 핵심 한 줄 (외워둘 것)
"AI로 콘텐츠를 대량 생성하되 '장면이 보이는 문장'만 통과시키는 2층 품질 게이트를 만들고, 콘텐츠·음원·삽화를 각각의 파이프라인으로 자동화해 웹/iOS/Android 단일 코드베이스로 배포한 오디오 학습 앱. 채점 정확도, 플랫폼별 포맷 버그, 배포 의존성 관리까지 직접 설계·해결."
