# [문서2] 네트워크 부가 설명 — 난이도 있는 개념 보충
> 문서1에서 한 번 더 파고들 가치가 있는 것들만 콤팩트하게.

---

## ① 왜 3-way인가 (2번/4번이 아니라)
- 통신은 쌍방향. 각자 "내 시퀀스 번호"를 상대가 ACK해줘야 그 방향이 성립.
- 2번: 서버가 클라의 수신 능력을 확인 못 함.
- 4번: 서버의 SYN과 ACK를 한 패킷(SYN+ACK)으로 합칠 수 있어 낭비.
- ISN이 랜덤인 이유: 옛 연결의 지연 패킷 혼입 방지 + 시퀀스 예측 공격 방지.

## ② TIME_WAIT vs CLOSE_WAIT (제일 헷갈림)
- TIME_WAIT: "먼저 끊은 쪽"이 마지막 ACK 후 2×MSL 대기.
  이유 ①마지막 ACK 유실 시 상대의 FIN 재전송을 받아줄 소켓 필요 ②옛 패킷이 새 연결에 섞이는 것 방지.
  → 커널이 시간 지나면 자동 정리.
- CLOSE_WAIT: 상대 FIN에 ACK만 하고 내 앱이 close() 미호출.
  → 앱이 안 닫으면 영구 잔존 = 코드 버그 신호.
- 감별: `ss -tan | awk '{print $1}' | sort | uniq -c`

## ③ 흐름제어 vs 혼잡제어 (누구를 배려?)
- 흐름제어: "받는 쪽"이 감당 못 할까 봐 → rwnd로 수신 버퍼 여유 광고.
- 혼잡제어: "길(네트워크)"이 막힐까 봐 → cwnd로 스스로 추측·조절.
- 실제 송신량 = 둘 중 작은 값.
- Zero Window = 흐름제어의 극단(버퍼 0). Window Update로 재개, 유실 대비 Zero Window Probe.

## ④ BDP (처리량 트러블슈팅의 핵심)
- BDP = 대역폭 × RTT = "파이프를 가득 채우려면 동시에 떠 있어야 할 데이터량".
- window < BDP면 대역폭이 남아돌아도 처리량이 안 남. → 장거리 대용량 전송에서 필수 점검.
- 옛 window는 16bit(64KB) → 부족 → Window Scaling 옵션으로 최대 1GB 확장.

## ⑤ CUBIC vs BBR (혼잡제어 현대판)
- CUBIC(리눅스 기본): "손실 = 혼잡"으로 간주. 손실 나면 cwnd 줄임. 유선 안정 링크에 좋음.
- BBR(구글): "손실 ≠ 혼잡" 가정. 실측 대역폭·RTT로 BDP 직접 추정. 손실 잦은 무선/장거리에서 우수(유튜브 사용).

## ⑥ HOL Blocking → HTTP/3 (인과 체인)
- TCP는 순서 보존 → 앞 세그먼트 유실 시 뒤 데이터가 버퍼에 있어도 앱에 못 올림(HOL).
- HTTP/2는 한 TCP에 여러 스트림 → 하나 막히면 전부 막힘.
- HTTP/3(QUIC)은 UDP 기반 + 스트림별 독립 순서 → HOL 해결. TLS 1.3 내장.

## ⑦ PMTUD 블랙홀 (ICMP 함부로 막으면 안 되는 이유)
- 경로 최소 MTU 탐색은 "너무 크면 라우터가 ICMP로 알려줌"에 의존.
- ICMP 전면 차단 시 큰 패킷은 드롭되는데 알림도 막혀 → 송신자는 영문 모르고 재전송만 하다 타임아웃.
- 증상: "작은 요청 OK, 큰 응답만 멈춤". → 대표적 PMTUD 블랙홀.

## ⑧ Nagle × Delayed ACK (숨은 40ms 지연)
- Nagle: 작은 조각을 ACK 올 때까지 모아 보냄(오버헤드 절감).
- Delayed ACK: 수신자가 ~40ms 모았다 ACK.
- 둘이 만나면 서로 눈치 보다 수십~수백 ms 지연. 실시간 앱은 TCP_NODELAY로 Nagle 끔.

## ⑨ HTTPS 심화 — RSA 키교환 vs ECDHE (킬러 포인트)
- RSA 방식: 클라가 세션키 생성 → 서버 공개키로 암호화해 전송 → 개인키로 복호화.
  약점: 트래픽 녹음 후 개인키 유출되면 과거 전부 복호화.
- ECDHE(1.3 강제): 세션키를 전송 안 함. 양쪽이 임시 키쌍 만들어 공개값만 교환 → 각자 같은 키 독립 계산(디피-헬만).
  개인키 유출돼도 과거 안전 = Forward Secrecy.
- 한 문장: "RSA는 세션키를 암호화해 보내고, ECDHE는 서로 계산해 전송 자체를 안 한다."

## ⑩ 그럼 인증서 개인키는 뭐에 쓰나
- 키 교환용이 아님. 신원 증명(서명)용.
- 서버가 ECDHE 공개값에 자기 개인키로 서명 → 클라가 인증서 공개키로 검증 → "진짜 그 서버" 확인.
- 이게 Forward Secrecy가 가능한 이유(키는 임시, 개인키는 서명 전용).

## ⑪ MITM은 왜 못 뚫나
- 공격자 인증서엔 신뢰받는 CA의 서명이 없음 → 브라우저 "신뢰 불가" 경고.
- CA 체인 검증이 MITM 차단의 핵심.
- 사내 프록시가 HTTPS를 까보는 건, 사내 Root CA를 직원 PC에 미리 심어서 합법화한 것.

## ⑫ DNS 조회 순서 (재귀 vs 반복)
- 캐시 계층: 브라우저 캐시 → OS 캐시 → hosts 파일 → 라우터 → ISP DNS.
- ISP DNS(재귀 리졸버)가 대신 Root → TLD(.com) → 권한 네임서버 순으로 반복 질의.
- 클라 입장은 "한 번 물으면 답 받음"(재귀), 리졸버는 여러 서버 순회(반복).

## ⑬ IP 고정 · MAC 홉마다 교체 (제일 헷갈림)
- 목적지 IP: 출발부터 도착까지 변하지 않음 (종단 식별자).
- 목적지 MAC: 매 홉마다 "바로 다음 장비의 MAC"으로 교체됨.
- 그래서 밖으로 나갈 때 ARP로 찾는 건 목적지 서버 MAC이 아니라 "게이트웨이(라우터)의 MAC".
- 최종 목적지 네트워크에 도착해서야 마지막 라우터가 실제 서버의 MAC을 ARP로 찾음.

## ⑭ 재귀 리졸버 vs 반복 질의 (DNS)
- 재귀(recursive): 클라 → 리졸버. "끝까지 알아내서 답만 다오".
- 반복(iterative): 리졸버 → Root/TLD/권한NS. "너 모르면 누구한테 물으라고만 알려줘"를 순차 수행.
- 즉 한 조회에 클라는 1번 묻지만, 리졸버는 뒤에서 여러 번 순회.

## ⑮ traceroute가 TTL을 악용하는 방식
- 데이터를 보내려는 게 아니라 "각 홉이 TTL=0에서 뱉는 오류 ICMP"를 수집하는 게 목적.
- TTL 1→첫 홉, 2→둘째 홉... 늘려가며 경로를 한 홉씩 밝힘.
- 중간 * * *는 그 홉이 ICMP 응답을 안 하는 것(꼭 장애 아님).

## ⑯ ping이 안 된다고 죽은 게 아니다
- 많은 서버·방화벽이 ICMP를 정책적으로 드롭.
- "ping 실패 = 서비스 다운"으로 단정 금지. 실제 서비스 포트(443 등)로 확인해야.
