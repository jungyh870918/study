# [문서17] 시스템 디자인 부가 설명 — 난이도 있는 개념 보충
> 문서16에서 한 번 더 파고들 가치가 있는 것들.

---

## ① CAP를 넘어 — PACELC
- CAP: 분할(P) 시 C vs A.
- PACELC: 분할(P) 시 A vs C, 아니면(Else) 지연(L) vs 일관성(C)까지 확장.
- 즉 분할 없는 평상시에도 "빠르게(낮은 지연) vs 정확하게(강한 일관성)" 트레이드오프가 있다.
- 예: DynamoDB=PA/EL(가용·저지연 우선), 전통 RDB=PC/EC(일관성 우선).

## ② 최종 일관성의 실제 (충돌 해결)
- 여러 노드에 동시 쓰기 → 충돌. 해결법:
  - Last-Write-Wins(타임스탬프 최신 우선, 단순하나 유실 위험)
  - 벡터 클록(인과관계 추적)
  - CRDT(수학적으로 병합 가능한 자료구조, 협업 편집)
- read-your-writes: 방금 쓴 걸 못 읽는 문제 → 쓴 직후엔 마스터로 라우팅.

## ③ 일관된 해싱 심화
- 가상 노드가 없으면 서버 3대일 때 부하가 33%씩 안 나뉘고 치우침 → 각 서버를 링에 100~200번 배치해 균등화.
- 서버 추가 시 이동하는 키 = 전체/서버수 (예: 3→4대면 약 1/4만 이동).

## ④ 큐 vs 스트림
- 메시지 큐(RabbitMQ): 작업 분배, 소비되면 사라짐.
- 이벤트 스트림(Kafka): 로그 보존, 여러 컨슈머가 각자 오프셋으로 재생 가능. 이벤트 소싱·재처리에 유리.
- 선택: 단순 작업 분배=큐, 이벤트 기록·재생·여러 소비자=스트림.

## ⑤ 캐싱 전략 4종
- Cache-aside(lazy): 앱이 캐시 확인→없으면 DB→캐시 채움. 가장 흔함.
- Write-through: 쓸 때 캐시+DB 동시. 읽기 빠름, 쓰기 느림.
- Write-back: 캐시에만 쓰고 나중에 DB. 빠르지만 유실 위험.
- Write-around: DB에만 쓰고 캐시는 읽을 때. 쓰고 안 읽는 데이터.

## ⑥ 데이터베이스 종류별 용도
- RDB(PostgreSQL/MySQL): 정합성·관계·트랜잭션.
- 문서(MongoDB): 유연 스키마, 중첩 데이터.
- 키-값(Redis/DynamoDB): 초고속 조회, 캐시·세션.
- 컬럼(Cassandra): 대용량 쓰기, 시계열.
- 그래프(Neo4j): 관계 탐색(친구의 친구).
- 검색(Elasticsearch): 전문 검색, 역색인.
- 시계열(InfluxDB): 메트릭·로그.

## ⑦ 낙관적 UI & 비동기 패턴
- 낙관적 업데이트: 서버 응답 전에 UI 먼저 반영(좋아요 즉시 표시)→실패 시 롤백. 체감 속도↑.
- 폴링 vs 롱폴링 vs WebSocket vs SSE: 실시간성 요구에 따라(채팅=WebSocket, 알림=SSE).

## ⑧ 관측성(Observability) 3축
- 로그(무슨 일이): 구조화 로깅+요청 ID.
- 메트릭(얼마나): QPS, 지연 p50/p99, 에러율.
- 트레이싱(어디서): 분산 추적(요청이 서비스들을 어떻게 통과).
- ★ p99가 평균보다 중요: 평균은 좋아도 상위 1% 사용자가 느리면 문제.
