## 1. 인덱스 원리 (B-Tree/B+Tree)
정의: 테이블 데이터를 정렬된 별도 구조로 미리 만들어 원하는 행을 빨리 찾는 자료구조. 책 뒤 색인.
인과: 없으면 Full Table Scan O(n), 있으면 트리 타고 O(log n). 100만 행: 풀스캔 100만 vs B-Tree ~20.

왜 B+Tree인가:
- 이유1 디스크 I/O 최소화: 디스크는 블록(4~16KB) 단위. 한 노드에 수백 키 담아 트리를 낮고 넓게(fan-out↑) → 100만 행도 3~4단계=디스크 접근 3~4번.
- 이유2 범위 검색: B+Tree는 실제 데이터가 리프에만, 리프끼리 링크드리스트 연결 → BETWEEN/ORDER BY 빠름(시작점 찾고 옆으로 쭉). InnoDB가 B+Tree 쓰는 결정적 이유.
- 해시 인덱스: =검색 O(1) 최고지만 범위·정렬 불가(순서 없음) → 특수 상황만.

클러스터형 vs 논클러스터형:
- 클러스터형: 인덱스 순서=실제 물리 저장 순서. 테이블당 1개(보통 PK).
- 논클러스터형(세컨더리): 별도 존재, 여러 개. 리프에 데이터 대신 위치(PK/포인터).
★ 세컨더리 2단계 조회(InnoDB): ①세컨더리에서 PK 찾고 ②PK로 클러스터 재조회해 데이터(back to table). 한 번 더 탐 → 살짝 느림 → 커버링 인덱스 최적화 등장(2강).

장단점(인덱싱 장단점):
- 장점: SELECT 극적 가속. WHERE/JOIN/ORDER BY/GROUP BY.
- 단점: 쓰기 저하(INSERT/UPDATE/DELETE 시 인덱스도 갱신) / 저장 공간 / 잘못 만들면 무용(카디널리티 낮은 컬럼).
- 한문장: "읽기를 O(log n)으로 빠르게 하는 대신 쓰기 비용·공간을 대가로. 읽기 많고 카디널리티 높은 컬럼에 선별적으로."

인덱스 안 타는 경우(실무 필수):
- 컬럼 가공: WHERE YEAR(date)=2024 ✗ → date>='2024-01-01' ✓
- 함수/연산: WHERE price*1.1>100 ✗
- LIKE '%검색' ✗ (LIKE '검색%' ✓)
- OR 조건(한쪽 인덱스 없으면 풀스캔)
- 카디널리티 낮음(옵티마이저가 풀스캔 선택)
- 암묵적 형변환(문자 컬럼에 숫자 검색)
- ★ 원칙: "컬럼은 그대로 두고 비교값을 가공하라." 컬럼 건드리면 인덱스 죽음.

풀스캔이 무조건 나쁜 건 아니다(인덱스 손익분기점):
- 세컨더리 인덱스 재조회 = 행마다 랜덤 I/O. 풀스캔 = 순차 I/O(디스크에 효율적).
- 읽을 행이 많아지면 인덱스 "골라읽기"가 손해 → 대략 전체의 5~20%(≈10%) 넘게 읽으면 풀스캔이 낫다.
- 정확한 수치 아님(행 크기/클러스터링/SSD 여부로 임계 변동. SSD는 랜덤 페널티 작아 임계↑).
- ★ 옵티마이저가 selectivity(조건이 몇 % 고르나)를 통계로 추정해 임계 넘으면 인덱스 있어도 일부러 풀스캔. = "카디널리티 낮으면 풀스캔 선택"의 메커니즘.
- 함정: "인덱스 있는데 왜 안 타?" 답이 종종 "옵티마이저가 그게 느리다고 판단". force index가 오히려 느려질 수 있음.
- 한문장: "인덱스는 소량 고를 때 유리, 읽을 비율이 임계(~10~20%) 넘으면 랜덤I/O 때문에 순차 풀스캔이 빠름."

## 2. 쿼리 최적화 (실행계획·인덱스 활용)

실행 계획(EXPLAIN):
정의: 옵티마이저가 세운 실행 전략. 튜닝의 출발점.
인과: SQL은 "무엇"만, "어떻게"는 옵티마이저가(선언형) → 선택에 따라 100배 차이 → EXPLAIN으로 들여다봄.
꼭 보는 것(MySQL):
- type: const>eq_ref>ref>range>index>ALL(풀스캔,최악). ALL이면 인덱스 안 탐 경고.
- key: 실제 사용 인덱스. NULL이면 안 씀.
- rows: 예상 훑을 행 수(적을수록 좋음).
- Extra: Using index(커버링,좋음)/Using filesort(정렬 별도작업,주의)/Using temporary(임시테이블,주의).
★ rows는 통계 기반 추정. 낡으면 오판. EXPLAIN ANALYZE는 실제 실행값.

커버링 인덱스:
정의: 필요한 모든 컬럼이 인덱스에 다 있어 테이블 재조회 불필요.
인과(1강 연결): 세컨더리 2단계(PK찾고 클러스터 재조회)를 없앰. SELECT 컬럼이 전부 인덱스에 있으면 인덱스만 읽고 끝. EXPLAIN Extra=Using index.
  인덱스(user_id,status): SELECT status WHERE user_id=5 → 커버링 / SELECT * → 재조회(커버링 아님)

복합 인덱스 & 왼쪽 접두사 원칙(제일 많이 틀림):
★ (a,b,c)는 왼쪽부터 연속으로 쓸 때만 탐:
  WHERE a=1 ✓ / a=1 AND b=2 ✓ / a=1 AND b=2 AND c=3 ✓(완전)
  WHERE b=2 ✗(a 건너뜀) / a=1 AND c=3 △(a까지만)
왜: a로 정렬→그 안에서 b→그 안에서 c. 전화번호부 성→이름 순, 이름만으론 못 찾음. 선두 컬럼 없으면 무용.
설계 원칙: 자주 쓰는 조건 앞에 / 등호(=)를 범위(>,<)보다 앞에(범위 뒤는 정렬 깨짐) / 카디널리티 높은 컬럼 앞.
한문장: "복합 인덱스는 왼쪽부터 연속으로 타고, 등호 앞·범위 뒤."

옵티마이저 & 통계:
정의: 여러 계획의 예상 비용(cost) 계산해 최소 선택(CBO). 근거=통계(행수,값 분포,카디널리티).
★ 통계 낡으면 오판: "10건일 것" 추정했는데 실제 100만 → 잘못된 계획(풀스캔/조인순서). "어제까진 빨랐는데 오늘 느림"의 흔한 원인. 대응 ANALYZE TABLE.
★ 그러나 통계 갱신은 양날의 검:
- plan regression: 통계 바뀌면 옵티마이저가 계획 재작성 → 잘 돌던 쿼리가 새 계획으로 오히려 느려질 수 있음(계획 퇴행). 핵심 쿼리에서 터지면 장애급.
- 갱신 부하: ANALYZE가 테이블 스캔 → 대형 테이블은 I/O·CPU 잡아먹어 바쁜 시간엔 업무 마비 위험.
- 실무 대응: 한가한 시간대에 / 큰 테이블은 샘플링 / 갱신 후 핵심 쿼리 계획 검증 / 중요 시스템은 계획 고정(baseline/hint).
- 태도: "최신이 항상 옳다"가 아니라 "최신화하되 영향을 검증한다".

조인 원리:
정의: 여러 테이블 연결. 내부 알고리즘 3종.
- Nested Loop: 각 행마다 다른 테이블 반복. 작은 테이블+인덱스 있을 때. OLTP 기본.
- Hash Join: 한쪽으로 해시 만들어 매칭. 큰 테이블+등호+인덱스 없을 때. 대량 분석.
- Sort Merge: 양쪽 정렬 후 병합. 정렬돼 있거나 범위 조인.
★ 조인 키(FK)에 인덱스 없으면 Nested Loop가 매번 풀스캔 = 재앙. 조인 키 인덱스 거의 필수. 옵티마이저가 driving table(먼저 읽을 테이블) 정하는데 통계 부정확하면 순서 틀어짐.
N:N: 중간 테이블(junction)로 분해. 학생-수업 N:N → student_course(student_id,course_id)로 1:N 두 개. 두 FK에 인덱스 걸어야 빠름.

## 3. 트랜잭션 · ACID · 격리수준 · 락 · MVCC

트랜잭션 & ACID:
정의: 더 못 쪼개는 하나의 논리적 작업 단위. 전부 성공/전부 실패(all-or-nothing). 예: 계좌 이체.
ACID: Atomicity(전부 or 롤백) / Consistency(제약조건 유지) / Isolation(서로 간섭 안 함) / Durability(커밋되면 죽어도 유지).
★ 구현: 원자성·지속성=WAL(Write-Ahead Log, 데이터 바꾸기 전 변경내역을 로그에 먼저 → redo/undo 복구). 격리성=락+MVCC.

격리 수준(제일 중요):
인과: 완벽 격리는 안전하나 느림(동시성 죽음) → 안전성vs성능 4단계로 선택.
이상 현상: Dirty Read(커밋 안 된 것 읽음) / Non-Repeatable(같은 행 두 번 값 다름) / Phantom(같은 조건 행 개수 다름).
4단계(낮음→높음):
- READ UNCOMMITTED: 다 발생
- READ COMMITTED: Dirty 방지
- REPEATABLE READ: Dirty+Non-Repeatable 방지 (InnoDB는 갭락으로 Phantom도 상당 방지)
- SERIALIZABLE: 다 방지
★ 기본값: PostgreSQL/Oracle=READ COMMITTED, MySQL(InnoDB)=REPEATABLE READ.

락(비관적):
- 공유 락(S): 동시 읽기 OK, 쓰기 막음
- 배타 락(X): 나만, 읽기·쓰기 다 막음
- 단위: 행 락<테이블 락(행이 동시성 좋음)
★ 데드락: A가 락1잡고 락2대기, B가 락2잡고 락1대기 → 영원 대기. DB가 한쪽 죽임. 예방: 락을 항상 같은 순서로 획득.

MVCC(5년차 킬러):
정의: 수정 시 덮어쓰지 않고 새 버전 생성 → 읽기와 쓰기가 서로 안 막음.
인과: 전통 락은 쓰는 중 읽기도 대기. MVCC는 읽는 쪽이 트랜잭션 시작 시점 스냅샷(옛 버전)을 봄 → 읽기가 쓰기 안 막고 쓰기가 읽기 안 막음. "읽기는 락 안 건다"가 정수. PG/InnoDB/Oracle 사용.
구현(InnoDB): 행에 버전(트랜잭션 ID), 옛 버전은 undo 로그. 읽을 때 볼 수 있는 버전 재구성(consistent read). 대가: 옛 버전 정리 필요(PG vacuum, InnoDB purge).
한문장: "수정 시 새 버전 만들어 읽는 쪽엔 스냅샷 → 읽기·쓰기 안 막음 → 락 없이 높은 동시성."

비관적 vs 낙관적 락:
- 비관적: 충돌 날 것 → 미리 락(SELECT FOR UPDATE). 충돌 많은 환경. 대기·데드락 위험.
- 낙관적: 충돌 드물 것 → 락 없이 커밋 시 version 컬럼으로 "남이 바꿨나" 확인, 바뀌면 재시도. 충돌 적은 환경.
실무: 재고 차감 등 충돌 잦으면 비관적, 일반 수정은 낙관적(버전 컬럼).

## 4. DB 아키텍처 (요소별 역할)

### 큰 틀: Database vs Instance
- Database = 디스크 위 파일 집합(데이터파일·컨트롤파일·리두로그·아카이브). 스스로 일 안 함(죽은 구조).
- Instance = DB를 움직이는 실행 주체(공유 메모리 + 백그라운드 프로세스).
- ★ 사용자는 파일에 직접 접근 못 함. 항상 Instance를 통과: User→Server Process→메모리→Background Process→Disk.

### 메모리 — 공유(SGA) vs 전용(PGA)
- SGA: 인스턴스 전체가 공유하는 메모리. 여러 세션이 같은 블록·SQL·메타정보 재사용.
- PGA: 서버 프로세스 전용. 정렬/해시/세션·커서 상태 등 개인 작업. 공유하면 꼬임.
- ★ 데이터는 SGA(버퍼 캐시)에서 읽고, 계산(sort/hash)은 PGA에서. "원본은 캐시, 계산은 작업장."
- 대가: SGA는 공유라 동시 접근 제어(latch/lock) 필요 → contention이 성능 병목의 원인.

### SGA 3대 구성
- Buffer Cache: 데이터파일 블록을 올려두는 작업장. 대부분 읽기·수정이 여기서 시작. 부족하면 physical read↑.
- Redo Log Buffer: 변경 이력을 잠시 담는 곳. "데이터보다 변경 이력을 먼저 안전하게."
- Shared Pool: SQL·실행계획 재사용(Library Cache) + 오브젝트 메타정보(Dictionary Cache).

### 버퍼 상태 3종 (Buffer Cache 내부)
- Free: 비어서 새 블록 읽어올 수 있음.
- Dirty: 메모리에서 변경됐지만 아직 디스크 미반영. 언젠가 DBWR가 내림.
- Pinned: 지금 누가 읽기/수정 중이라 못 밀어냄.
- LRU: 캐시가 유한 → 오래 안 쓴 블록부터 밀어냄(최근 쓴 건 또 쓸 확률 높음).

### 백그라운드 프로세스 5대 (역할 분담)
- DBWR: dirty buffer를 데이터파일에 내림(지연 쓰기). 이 덕에 commit이 빠름.
- LGWR: redo를 리두 로그 파일에 먼저 기록. ★ commit 순간 가장 중요한 프로세스(DBWR 아님).
- CKPT: "데이터파일이 redo의 어느 시점까지 따라왔나" 기준점 기록 → 복구 시작점 축소.
- SMON: 인스턴스 장애 복구(redo로 roll forward → commit 안 된 건 undo로 roll back).
- PMON: 죽은 세션/프로세스의 자원(PGA 등) 회수.

### WAL — 로그 먼저, 데이터 나중 (핵심 원리)
1. UPDATE 발생 → 2. Redo Log Buffer 먼저 기록 → 3. Buffer Cache 블록 변경 → 4. COMMIT 시 LGWR가 Redo 파일 기록 → 5. DBWR가 나중에 데이터파일 반영.
- 왜: 데이터파일 갱신은 랜덤 write라 느림, redo는 append라 빠름. 정합성은 redo로 보장, 실제 쓰기는 미룸.
- ★ Fast Commit: commit = "데이터파일에 다 썼다"가 아니라 "복구 가능한 redo가 디스크에 안전히 기록됐다".
- 복구: 장애 시 checkpoint 이후 redo로 roll forward → commit 안 된 것 undo로 roll back.

### 프로세스 vs 스레드 모델
- 프로세스 기반(Oracle/PostgreSQL): 유저 1명 = 서버 프로세스 1개(Dedicated). 세션·커서·트랜잭션 상태 분리, PGA가 프로세스 전용이라. 격리 좋음, 생성 비쌈.
- 스레드 기반(MySQL): 커넥션마다 스레드. 가볍고 빠름, 하나 크래시가 전체 영향 가능.
- ★ Too Many Connections: 유저 1000명=프로세스 1000개=PGA 1000개 → 메모리 폭증·context switching↑ → 커넥션 풀 필요.

### 커넥션 풀 (요약)
- 정의: 커넥션을 미리 N개 만들어 빌려주고 반납. 유저 수와 DB 커넥션 수를 분리해 DB 보호.
- ★ 크기: 작으면 대기, 크면 DB 프로세스 폭증. 느린 쿼리·누수·롱 트랜잭션이 풀 고갈 유발. WAS에서 max/min/timeout/validation 설정.

### DB 기동 3단계 (역할이 다름)
- NOMOUNT: parameter file 읽어 SGA·백그라운드 프로세스 기동. 인스턴스만 생김. 파일 안 붙음.
- MOUNT: control file 열어 데이터·리두 파일의 "지도"(위치·checkpoint 메타) 확보. 데이터 접근 불가. 복구 판단·파일 이동이 여기서.
- OPEN: 데이터파일 열고 정합성 확인(필요시 recovery). 사용자 SQL 가능. ★ "RAM에 다 올림"이 아니라 "서비스 가능한 정합성 상태".
- 순서 불가역: parameter→control→datafile. 실패 단계로 원인 분리(NOMOUNT=파라미터/메모리, MOUNT=컨트롤파일, OPEN=데이터파일/복구).
- Shutdown은 역순(close→unmount→종료). 실무 기본 shutdown immediate.

한문장: "Instance가 파일을 대신 다루고, SGA(공유)·PGA(전용)로 메모리를 나누며, LGWR가 redo를 먼저 써 정합성을 보장하고 DBWR가 나중에 데이터를 내린다. 기동은 NOMOUNT→MOUNT→OPEN으로 점검하며 올린다."

## 5. 데이터 모델링 (정규화·관계·자료형)

정규화:
정의: 중복 제거해 이상 현상 막는 설계. 각 데이터가 한 곳에만.
이상 현상 3종: 삽입 이상(불필요 데이터 없이 못 넣음)/갱신 이상(중복 하나만 고쳐 불일치)/삭제 이상(지우면 딸린 정보 소실).
단계(보통 3NF까지): 1NF 원자값 / 2NF 부분 종속 제거 / 3NF 이행 종속 제거.
암기: "키에, 전체 키에, 오직 키에만 종속되게."

반정규화(5년차 핵심):
정의: 의도적으로 중복 허용해 조회 성능↑.
인과: 정규화하면 조인 많아져 느림. 자주 함께 읽는 데이터를 중복 저장해 조인 감소. 예: 주문에 user_name 중복.
★ 트레이드오프: 얻음=읽기 속도, 잃음=쓰기 복잡+정합성 위험(갱신 시 여러 곳 동기화).
원칙: "정규화로 설계하고, 병목 증명되면 선택적으로 반정규화." 처음부터 반정규화 X.

관계(1:1, 1:N, N:N):
- 1:1: 드묾(보안·분리).
- 1:N: 가장 흔함. N쪽에 FK(주문에 user_id).
- N:N: 직접 표현 불가 → 중간 테이블로 1:N+1:N 분해.
★ 중간 테이블: enrollment(student_id, course_id, 신청일, 성적). 두 FK 인덱스, 보통 복합 PK. 관계 자체의 속성(신청일/성적)을 담을 수 있는 게 진짜 이점.
조인 키(FK)에 인덱스 필수.

키:
- PK: 유일 식별, NULL 불가. 보통 클러스터 인덱스.
- FK: 다른 테이블 PK 참조. 참조 무결성 보장.
★ 자연 키 vs 대리 키: 자연(주민번호/이메일, 바뀔 수 있고 노출 위험) vs 대리(auto_increment/UUID, 의미 없고 안 변함=실무 기본).
- auto_increment(순차, 인덱스 효율 좋지만 예측가능·분산 충돌) vs UUID(전역 유일·분산 유리, 랜덤이라 인덱스 삽입 저하) → 순차 UUID(v7)로 절충.

자료형과 용량:
정의: 타입 선택이 공간·성능·정합성 직결. "대충 VARCHAR(255)" 금지.
- 정수: TINYINT(1B)/SMALLINT(2B)/INT(4B)/BIGINT(8B). 범위 보고 최소로. 21억 넘으면 BIGINT.
- 실수: FLOAT/DOUBLE(오차) vs DECIMAL(정확). ★ 금액은 반드시 DECIMAL(FLOAT는 반올림 사고).
- 문자: CHAR(고정, 국가코드) vs VARCHAR(가변, 대부분) / TEXT(대용량, 인덱스 제약).
- 날짜: DATE/DATETIME(8B)/TIMESTAMP(4B, UTC 변환·2038 한계).
- ★ TIMESTAMP(UTC 저장·타임존 자동변환·절약, 2038 문제) vs DATETIME(범위 넓음·타임존 없음).
- 불리언: MySQL은 TINYINT(1). JSON: 유연하나 인덱싱·정합성 약함(남용 주의).
★ 왜 아끼나: 행 작을수록 한 페이지에 더 많은 행→같은 I/O로 더 읽음→버퍼 풀 효율↑. 인덱스 컬럼 작을수록 빠름.
한문장: "타입은 필요 범위의 최소로, 금액은 DECIMAL, 날짜는 타임존 요구로 TIMESTAMP/DATETIME 구분. 행이 작아야 페이지·인덱스 효율↑."

스키마 설계 대표 케이스:
- 이력/변경 추적: 현재값+이력 테이블 분리, 또는 유효기간(valid_from/to).
- 상태 관리: 상태 컬럼+전이 로그(pending→paid→shipped).
- 계층 구조: 인접 리스트(parent_id) vs 경로 열거 vs 클로저 테이블.
- 소프트 삭제: deleted_at 플래그(복구·감사).
- 다형 관계: target_type+target_id(댓글이 게시글·사진 등에).
- 감사: created_at/updated_at/created_by 공통 컬럼.
★ 원칙: "정규화 기본, 읽기 패턴 보고 인덱스·반정규화 추가. 키는 대리키, 금액은 DECIMAL, 삭제는 대개 소프트 삭제."
