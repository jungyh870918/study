# [문서19] 시스템 디자인 확인 질문 — 핵심 점검용
> 이해했다면 1~2문장으로 바로 답할 수 있어야 함.

---

## 1. 접근 & 기본
- 시스템 디자인 문제를 받으면 첫 단계는? (요구사항 명확화)
- 규모 추정에서 무엇을 계산? 왜 읽기:쓰기 비율이 중요?
- 수직 확장과 수평 확장 차이? 수평의 전제 조건은? (무상태)
- 왜 상태를 외부화해야 하나?

## 2. 분산 핵심
- CAP 정리를 한 문장으로? 분할 시 무엇을 선택?
- CP와 AP 시스템의 예와 용도?
- 최종 일관성이란? 어떤 데이터에 허용 가능?
- 강한 일관성이 꼭 필요한 데이터 예시?
- SPOF를 어떻게 찾고 제거하나?
- 일관된 해싱이 왜 필요? 단순 hash%N의 문제는?
- 가상 노드는 왜 두나?

## 3. 패턴
- 메시지 큐가 주는 3가지 이점?
- at-least-once와 멱등성의 관계?
- 멱등성 키는 어디에 쓰나?
- Rate Limiting 대표 알고리즘?
- DB 확장 순서? 왜 샤딩이 최후인가?
- 마이크로서비스가 항상 옳지 않은 이유?

## 4. 심화
- CAP와 PACELC 차이?
- 캐싱 전략 4종과 각 특징?
- 큐와 스트림(Kafka) 차이?
- 관측성 3축? 왜 p99가 평균보다 중요?
- read-your-writes 문제와 해결?

## 4-B. 대량 트래픽 핸들링
- 트래픽 대응의 핵심 원칙? (은탄환 없음, 계층별 방어선)
- 방어선을 바깥→안으로 나열하면?
- 왜 바깥에서 막는 게 유리한가?
- 캐시 무효화를 안 하면? Cache Stampede란? 대응은?
- Hot Key 문제와 대응?
- 수평 확장의 전제 조건? (무상태)
- 오토스케일 두 층? (HPA / 노드)
- 스케일 아웃/인의 속도는 왜 다르게? 예측 이벤트 대응은? (예열)
- 읽기는 캐싱으로 막는데 쓰기는 무엇으로? (큐)
- DB 확장 순서? 왜 샤딩이 최후?
- 모니터링 3축? 왜 p99가 평균보다 중요?
- 반드시 보는 지표 3~4개?
- Circuit Breaker/Graceful Degradation/Bulkhead 각각?

## 종합 (설계 문제)
- "URL 단축기를 설계해보라." (해싱, 캐싱, 리다이렉트, 규모)
- "실시간 채팅을 설계해보라." (WebSocket, Pub/Sub, 순서, presence)
- "뉴스 피드를 설계해보라." (fan-out, 캐싱, 랭킹)
- "결제 시스템을 설계해보라." (멱등성, 큐, 정합성, 웹훅)
- 각 설계에서 트레이드오프를 소리 내어 설명할 수 있는가?
- "트래픽이 평소의 100배로 급증했다. 계층별로 어떻게 대응하나?"
