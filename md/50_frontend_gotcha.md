# [문서] 허 찌르는 질문 대비 (경험·버전·선택 트레이드오프)
> 지식이 아니라 "실제 경험·선택"을 묻는 질문. 지어내지 말고 실제 기반으로. 모르면 정직하게, 아는 건 뻣뻣하게.
> ★ 경력직 답변 태도(문서0)와 세트: 개념은 아는 대로, 경험은 실제만.

---

## 1. "최근에 써본 Next 버전 몇이에요?"
- 사실 확인: 2026년 기준 Next 16이 stable(2025년 10월 출시), 16.2.x가 현재 안정 라인. 15는 유지보수 LTS.
- ★ 답변 원칙: 실제 쓴 버전을 말한다. 병원결제 프로젝트가 Next 15(App Router)였으면 "15로 App Router 기반 결제 플랫폼을 만들었습니다"로. 안 써본 최신 버전을 안다고 하지 말 것.
- 버전 감각 보여주기(안 써봤어도 흐름은 알기):
  - 13: App Router 등장(서버 컴포넌트).
  - 14: 서버 액션 안정화.
  - 15: React 19, async request API, Turbopack 안정, fetch 캐싱 기본값 변경.
  - 16: Turbopack 기본 번들러(webpack 대체), React Compiler 안정, next lint 제거(ESLint 직접 실행), 라우팅/프리페치 최적화.
- ★ "15까지 써봤고 16은 Turbopack이 기본이 되고 캐싱이 명시적으로 바뀐 걸로 압니다" 식으로 "쓴 것+아는 것"을 구분해 답하면 정직하면서 감각 있어 보임.

## 2. "상태관리 툴 뭐까지 써봤어요?"
- ★ 실제 쓴 것만. 프로젝트 기반으로:
  - 서버 상태: React Query(TanStack) — 캐싱·refetch.
  - 클라 전역: Redux Toolkit 또는 Zustand.
  - 로컬: useState / useReducer, 저빈도 전역: useContext.
- 안 써봤으면 "Redux는 써봤고 Zustand는 안 써봤는데 더 가벼운 대안으로 압니다"로 구분.
- ★ 후속 대비: "왜 그걸 골랐나?" → "서버 데이터라 React Query로 캐싱·동기화를 맡기고, 클라 전역(모달·필터)만 가벼운 Zustand로 나눴습니다." (서버상태/클라상태 분리 원칙, 문서26·46).
- 함정 질문: "Redux 없이 Context로 전역 못 하나?" → "가능하지만 Context는 값 바뀔 때 구독 컴포넌트가 다 리렌더돼서 자주 바뀌는 상태엔 부적합. 그래서 저빈도(테마·인증)만 Context, 자주 바뀌는 건 전용 도구."

## 3. "왜 그 기술을 골랐어요?" (선택 이유)
- ★ "남들이 써서"는 최악. 트레이드오프로 답한다.
- 예: "왜 Next?" → "SSR·SEO·라우팅·이미지 최적화가 필요했고, React만으론 다 조립해야 해서. 풀스택이라 백엔드도 한 코드베이스로." (문서46 C8)
- 예: "왜 Prisma?" → "SQLite↔PG 무변경, 타입 안전. 단 복잡 쿼리는 raw도 병행." (문서48)
- 예: "왜 Zustand(또는 Redux)?" → 규모·보일러플레이트·팀 친숙도 트레이드오프.
- 정직 카드: "사실 이건 회사에서 이미 쓰던 스택이라 따라간 부분도 있습니다"도 솔직한 답(무리한 정당화보다 나음).

## 4. "그거 안 써봤죠?" / 모르는 것 찌르기
- ★ 지어내지 말 것. "그건 직접 안 써봤습니다. 다만 개념은 ~로 이해하고 있습니다"로 정직+지식.
- 개념 질문이면 경험 언급 없이 아는 대로 뻣뻣하게(문서0). "잘 모르겠습니다" 고백보다 아는 만큼.
- 절대 금지: 안 써본 걸 써봤다고 → 후속 질문에서 무너짐. 한 번 거짓이 드러나면 전체 신뢰 붕괴.

## 5. 자주 나오는 "허 찌르기" 유형별 대비
- "이거 최신 버전에선 바뀐 거 아세요?" → 모르면 "그 부분은 최신 변경을 못 따라갔습니다. 확인해보겠습니다." (모르는 걸 인정하는 게 감점 아님)
- "그 라이브러리 단점은?" → 장점만 말하면 의심받음. 반드시 트레이드오프도: "React Query는 서버상태엔 좋지만 클라 전역엔 안 맞고, 러닝커브가 있습니다."
- "직접 밑바닥부터 구현할 수 있어요?" → "라이브러리로 했지만 원리는 ~입니다"(예: JWT 검증, 상태관리 구독). 원리를 알면 OK.
- "왜 A 대신 B 안 썼어요?" → 둘 다 알고 트레이드오프로 판단했음을 보이기. "B도 고려했는데 A가 ~라서."
- "이 코드 왜 이렇게 짰어요?" → 당시 판단 근거를 정직하게. 더 나은 방법 알면 "지금이라면 ~로 하겠습니다"(성장 서사).
- "성능 최적화 해봤어요?" → 실제 한 것만. 측정 없이 "최적화했다"는 위험. "N+1을 eager loading으로, 이미지를 next/image로" 등 구체적으로.

## 6. 버전·트렌드 감각 (안 써봤어도 흐름은)
- React: 19가 최신 라인(Server Components, Actions, use 훅, useEffectEvent). React Compiler로 자동 memoization.
- Next: 16이 stable(Turbopack 기본, React Compiler 안정).
- 상태관리 트렌드: Redux → Redux Toolkit(보일러플레이트↓) → Zustand/Jotai(경량) → 서버상태는 React Query가 표준.
- 스타일: CSS Modules → styled-components(CSS-in-JS) → Tailwind(유틸리티, 현재 우세) + shadcn/ui.
- ★ "최신을 다 안 써봤어도 흐름과 이유는 안다"를 보이면 됨. 트렌드의 방향(왜 이렇게 바뀌었나)이 핵심.

## 7. 답변 태도 요약 (문서0과 연결)
- 경험 질문 = 실제만, 지어내지 말 것. 안 한 건 "안 했다".
- 개념 질문 = 경험 언급 없이 아는 대로 뻣뻣하게. 고백 금지.
- 선택 질문 = 트레이드오프로. "남들이 써서"는 금지.
- 모르는 것 = 정직하게 인정 + 아는 부분은 말하기. 판단은 면접관 몫.
- 항상 후속 대비: 한 번에 다 쏟지 말고 결론 먼저, 후속에서 심화.

## 핵심 요약
허 찌르는 질문은 지식이 아니라 "실제 경험·선택·트레이드오프"를 본다. 실제 쓴 버전·툴만 말하고(Next 15/React Query/Redux 등 프로젝트 기반), 안 써본 건 정직하게 인정하되 개념·흐름은 아는 대로 답한다. 선택 이유는 "남들이 써서"가 아니라 트레이드오프로, 단점도 함께. 버전 트렌드는 안 써봤어도 방향(왜 바뀌었나)을 알면 감각 있어 보인다. 지어내면 후속에서 무너지니 절대 금지.
