# [문서10] DB 실전편 — 실무·경험 기반
> 이론이 아니라 "실무에서 어떻게 쓰고, 면접에서 경험을 어떻게 말하나".
> 각 항목: 개념 → 실무 포인트 → 경험 서술.

---

## 1. N+1 문제 (ORM 최대 함정)
정의: 목록 N개 조회 후 각 항목의 연관 데이터를 N번 더 쿼리 → 총 1+N번. ORM 지연 로딩(lazy) 때문에 자기도 모르게 발생.
예: 게시글 10개(1번) → 작성자 각각(10번) = 11번. 1000개면 1001번.
해결: 즉시 로딩/조인으로 한 번에.
- JPA: fetch join, @EntityGraph
- Django: select_related(1:1,N:1) / prefetch_related(N:N,1:N)
- TypeORM/Sequelize: relations / include
- 핵심: "한 번의 조인 or IN 쿼리로 묶는다."
★ 로컬(데이터 적음)에선 안 보이다 운영에서 터짐. 쿼리 로깅 켜서 실제 쿼리 수 확인이 방어책.
경험: "목록 API가 느려 쿼리 로그 보니 N+1이었고, fetch join으로 11번→1번으로 줄였다."

## 2. 슬로우 쿼리 튜닝 흐름
1) 슬로우 쿼리 로그로 느린 쿼리 식별(임계 시간)
2) EXPLAIN으로 실행 계획 확인(type=ALL 풀스캔인가, 인덱스 타는가)
3) 인덱스 추가/수정 or 쿼리 재작성(컬럼 가공 제거)
4) 재측정
5) 필요시 반정규화·캐싱·페이지네이션
경험: "슬로우 로그로 잡은 쿼리가 YEAR(date) 가공 때문에 인덱스를 못 타서, 범위 조건으로 바꿔 풀스캔을 없앴다."

## 3. 복제(Replication) & 읽기/쓰기 분리
정의: 마스터(쓰기)-레플리카(읽기) 복제. 읽기 트래픽을 레플리카로 분산.
인과: 읽기 압도적으로 많은 서비스에서 쓰기는 마스터 1대, 읽기는 여러 레플리카로. WAL/binlog를 레플리카가 재생해 동기화(LSN).
★ 복제 지연(replication lag): 마스터에 쓴 게 레플리카 반영까지 시간차 → "방금 쓴 걸 바로 읽었는데 없음"(read-your-writes). 중요한 직후 조회는 마스터로.
경험: "읽기 부하가 커서 레플리카로 조회를 분산했고, 결제 직후 조회는 lag 때문에 마스터로 라우팅했다."

## 4. 샤딩 & 파티셔닝
파티셔닝: 한 DB 안에서 큰 테이블을 기준(날짜/범위/해시)으로 조각. 필요한 파티션만 스캔(프루닝).
샤딩: 데이터를 여러 DB 서버에 수평 분할. 한 대로 감당 안 될 때.
- ★ 샤드 키 선택이 핵심: 잘못 고르면 특정 샤드 몰림(hotspot).
- 대가: 샤드 간 조인·트랜잭션 어려움, 재분배 복잡 → "정말 필요할 때만."
경험: "로그 테이블이 커져 월 단위 파티셔닝으로 조회·삭제를 파티션 단위로 처리했다."

## 5. 데드락 실전 대응
- 데드락 로그로 충돌한 쿼리 확인
- 락 획득 순서 통일(순환 제거)
- 트랜잭션 짧게
- 인덱스로 락 범위 축소(범위 넓으면 갭 락으로 충돌↑)
경험: "두 배치가 반대 순서로 테이블을 잠가 데드락 → 락 순서를 통일해 해결."

## 6. ORM 함정 모음
- N+1(위)
- SELECT *: 불필요 컬럼까지 → 필요 컬럼만 지정
- 트랜잭션 경계 모호 → 롱 트랜잭션 유발
- 대량 처리에 ORM: 수만 건 반복 저장 느림 → bulk insert/raw query
- 지연 로딩 남발 → N+1
- 원칙: "ORM은 편하지만 나가는 SQL을 항상 의식하라. 로깅으로 검증."

## 7. 대용량/성능 실무 기타
- 페이지네이션: OFFSET 큰 값은 느림(앞의 것 다 세고 버림) → 커서 기반(WHERE id > 마지막id)이 효율적.
- 벌크 연산: 반복 INSERT 대신 배치 INSERT, UPDATE는 조건 묶어서.
- 커넥션 풀 고갈: 느린 쿼리·누수·롱 트랜잭션 확인(문서6 4강).
- 캐싱: 자주 읽고 덜 바뀌는 데이터는 Redis(cache-aside), 무효화 주의(네트워크 문서5).
- 인덱스 재구성: 대량 삭제·갱신 후 인덱스 단편화 → 재빌드 고려.

## 8. 서브쿼리·조인 남발·쿼리 지옥
왜 지옥: 조인/서브쿼리 자체는 문제 아님. 옵티마이저가 감당 못 하는 복잡도가 문제.
- 조인 남발: 5~6개+ 조인이면 옵티마이저가 조인 순서 경우의 수 폭증→나쁜 계획. 조인 키 인덱스 없으면 재앙.
- 중첩 서브쿼리: 상관 서브쿼리(외부 값 참조)는 행마다 반복 실행→O(n²)처럼.
- 가독성 지옥: 5단 중첩은 6개월 뒤 나도 못 읽음.
해법:
- 상관 서브쿼리→JOIN(옵티마이저 유리).
- 깊은 중첩→CTE(WITH)로 평탄화, 단계별 이름.
- 너무 복잡하면 쿼리 쪼갬(앱에서 나눔/임시 테이블).
- 반정규화·집계 테이블(매번 무거운 조인 대신 미리 계산).
- EXPLAIN으로 검증.
★ 원칙: "조인·서브쿼리 많다고 나쁜 게 아니라 옵티마이저가 이해 못 하거나 사람이 못 읽으면 나쁘다. 인덱스로 옵티마이저 돕고, CTE로 사람 돕고, 정 안 되면 쪼갠다."

## 9. 복잡도 경험을 면접에서 녹여내는 법
나쁜 예(숫자 자랑): "8개 테이블 조인 짰습니다" → 못 읽는 쿼리 자백일 수도.
좋은 예(문제→판단→결과):
"정산 리포트 6테이블 조인이 3초 넘어서, EXPLAIN 보니 조인 키 인덱스 없어 풀스캔 겹침. 인덱스 걸고 매번 계산하던 집계를 일 단위 집계 테이블로 반정규화해 300ms로. 대신 갱신 배치를 새벽에 돌리는 트레이드오프 감수."
구조: ①구체 상황 ②진단 도구(EXPLAIN) ③원인 ④해결 ⑤트레이드오프 인식.
★ 핵심: "복잡한 걸 다뤄봤다"가 아니라 "복잡한 걸 단순하게 만들 줄 안다". 진짜 시니어는 복잡한 쿼리를 자랑 안 하고 안 복잡하게 만든 이야기를 함.

## 10. Python MySQL 라이브러리
- mysql-connector-python: 공식, 순수 파이썬.
- PyMySQL: 순수 파이썬, 가볍고 널리 씀.
- mysqlclient: C 확장, 빠름(컴파일 의존성).
- SQLAlchemy: ORM/쿼리 빌더(위 드라이버 위). Core는 SQL에 가깝게, ORM은 객체 매핑.
★ 실무:
- 커넥션 풀 사용(SQLAlchemy/DBUtils).
- ★ 파라미터 바인딩 필수: execute("...WHERE id=%s", (uid,)). f-string 금지(SQL 인젝션). 타협 불가.
- with 문으로 커서·커넥션 관리(안 닫으면 누수).

## 11. NestJS에서 ORM 대신 생 SQL
언제: ORM 쿼리가 비효율(N+1/불필요 조인), 복잡 집계·윈도우 함수·DB 고유 기능, 성능 critical 핫패스.
TypeORM:
- dataSource.query('SELECT ... WHERE age > $1', [18]) — 완전 생 SQL(파라미터 바인딩).
- createQueryBuilder — SQL에 가깝지만 타입 안전.
★ 판단: 기본은 ORM(생산성·타입·유지보수), 필요한 곳만 생 SQL(성능·복잡). 전부 생 SQL=역행, 전부 ORM=성능 포기. 파라미터 바인딩 필수(인젝션). "종교 전쟁 아니라 케이스별 도구 선택."
