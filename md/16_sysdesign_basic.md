# [문서16] 시스템 디자인 기본 — 면접 실전용
> 핵심은 개념 암기가 아니라 "트레이드오프를 저울질하고 선택을 정당화".
> 빌딩 블록은 네트워크·DB편에서 이미 다룸. 여기선 "언제 무엇을 조합하나".

---

## 1. 접근 프레임워크 (구조화된 순서)
개방형 문제를 이 순서로 끊는다:
1. 요구사항 명확화: 기능(핵심 기능?)+비기능(규모/지연/가용성/일관성). ★ 바로 그리지 말고 질문으로 범위 좁히기.
2. 규모 추정(back-of-envelope): DAU, QPS, 저장 용량, 대역폭, 읽기:쓰기 비율(대개 읽기 압도)→캐싱/샤딩/CDN 결정.
3. API 설계: 핵심 엔드포인트(계약). REST/gRPC.
4. 데이터 모델 & 저장소: RDB vs NoSQL, 스키마, 인덱스(DB편).
5. 고수준 설계(HLD): 클라→LB→서버→캐시→DB, 데이터 흐름.
6. 상세 설계(Deep Dive): 병목 컴포넌트 깊게(면접관 유도).
7. 병목 식별 & 확장: SPOF 제거, 스케일링, 모니터링.
★ 사고 과정을 소리 내어 설명. "왜 이 선택인지" 계속 말하기. 침묵하며 그리지 말 것.

## 2. 스케일링 (Vertical vs Horizontal)
- 수직(scale-up): 사양 업그레이드. 간단하나 한계·SPOF.
- 수평(scale-out): 대수 늘림. 무한 확장, 분산 복잡. 로드밸런서 필요.
- ★ 상태 외부화(세션을 Redis로)해야 수평 확장 가능. 대규모의 답은 수평.

## 3. 핵심 빌딩 블록 (기존 편 통합)
- 로드밸런서: L4(IP/포트) vs L7(URL/헤더). RR/least_conn/IP해시. (네트워크편)
- 캐싱: CDN(정적)/Redis(동적·세션)/앱 캐시. cache-aside, 무효화 주의. (네트워크편)
- DB 확장: 복제(읽기 분산)+샤딩(쓰기 분산). (DB편)
- 메시지 큐: Kafka/RabbitMQ로 비동기·완충·디커플링(backpressure). (연상사전)

## 4. CAP 정리
정의: 분산은 일관성(C)·가용성(A)·분할내성(P) 중 둘만. 분할(P)은 불가피 → 분할 시 C냐 A냐 선택.
- CP(일관성 우선): 분할 시 일부 막아서라도 정합성. 금융·결제. (HBase, MongoDB 기본)
- AP(가용성 우선): 분할 시에도 응답, 일시 불일치 허용. SNS 피드·좋아요. (Cassandra, DynamoDB)
★ 실제론 스펙트럼. 최종 일관성(eventual): "결국 일치". 대규모의 현실적 타협. = DB 복제 지연.

## 5. 일관성 vs 가용성 트레이드오프 (핵심 사고)
- 강한 일관성 필요: 계좌 잔액, 재고, 예약 → 틀리면 사고.
- 최종 일관성 허용: 좋아요 수, 조회수, 팔로워 수 → 잠깐 틀려도 됨.
- ★ "모든 걸 강하게"는 느리고 비쌈. 기능별로 나눠 판단하는 게 5년차.

---

## 6. SPOF 제거 & 고가용성
정의: SPOF=하나 죽으면 전체 멈춤. HA 첫걸음은 SPOF 제거.
- 이중화: 모든 계층 복수(LB 2대 active-standby, DB 복제)
- failover: 주 노드 죽으면 대기 승격(마스터 장애→레플리카 승격)
- 헬스체크: LB가 죽은 서버 감지해 제외
★ "이 컴포넌트 죽으면?"을 모든 박스에 질문. "전체 중단"이면 이중화 필요.

## 7. 일관된 해싱(Consistent Hashing)
정의: 서버 추가/제거 시 재배치 최소화. 캐시·샤딩·분산 저장 핵심.
인과: 단순 hash%N은 N 바뀌면 거의 전부 재배치→캐시 대량 무효화. 일관된 해싱은 원형 링에 배치→인접 구간 키만 이동.
- 가상 노드: 서버를 링에 여러 번 배치해 부하 고르게(hotspot 방지).
용도: 분산 캐시, Cassandra/DynamoDB 파티셔닝. "서버 늘려도 재배치 최소".

## 8. 메시지 큐 & 비동기 (심화)
정의: 생산자-소비자를 큐로 분리(비동기·완충·디커플링).
- 완충: 급증을 큐가 흡수, 소비자는 감당 속도로(backpressure)
- 디커플링: 생산자는 소비자 상태 몰라도 됨(장애 격리)
- 비동기: 무거운 작업(메일/이미지) 분리→응답 빠름
★ 전달 보장: at-most-once(유실 가능)/at-least-once(중복 가능→멱등성으로, 실무 기본)/exactly-once(이상적·비쌈). Dead Letter Queue로 무한 재시도 방지.

## 9. 멱등성(Idempotency)
정의: 같은 요청 여러 번 실행해도 결과가 한 번과 같음.
인과: 네트워크 불안정→재시도 필연. 재시도로 결제 두 번=재앙 → 멱등성 키로 "이미 처리?" 확인. 결제/주문/at-least-once 큐 필수.

## 10. 속도 제한(Rate Limiting)
- 토큰 버킷: 일정 속도로 토큰 채우고 요청마다 소비. 버스트 허용(가장 흔함).
- 누출 버킷: 일정 속도로만 처리(부드러운 유출).
- 고정/슬라이딩 윈도우: 시간 창 카운트.
- 분산 환경은 Redis로 카운터 공유.

## 11. DB 확장 결정 트리 (통합)
- 읽기 병목 → 복제+캐싱
- 쓰기 병목 → 샤딩
- 관계 복잡·정합성 → RDB 유지
- 유연·초대용량·단순 조회 → NoSQL
- 분석/집계 → OLAP/DW
★ 순서: 캐싱→읽기 복제→인덱스 최적화→그래도 안 되면 샤딩. 샤딩은 조인·트랜잭션 잃어 최후 수단.

## 12. 마이크로서비스 vs 모놀리스
- 모놀리스: 단일 코드베이스. 초기 간단·네트워크 호출 적음. 부분 확장 어렵고 커지면 관리↑.
- 마이크로서비스: 독립 배포 서비스들. 확장·장애 격리·독립 배포. 복잡도·네트워크 오버헤드↑.
★ 선택: 크고 빠르게 진화=마이크로서비스, 작은 팀·초기=모놀리스. "항상 마이크로서비스"는 착각. 분산 복잡성(네트워크 실패·정합성·분산 트랜잭션) 감당 가능할 때만.
